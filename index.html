UX Writing
Guidelines

디지털 정부서비스 UI·UX 가이드라인(2025.08) 기반. 일관된 언어와 친근하면서도 신뢰감 있는 문구로 사용자 경험을 높입니다.

필수 반드시 준수
권장 사용성 향상
우수 최고 수준
KWCAG 2.2 · WCAG 2.1 AA
Core Principles 01

언어 원칙

모든 UI 문구 작성 시 따라야 하는 기본 원칙입니다.

6가지 핵심 원칙
  • 1사용자 중심 — 시스템이 아닌 사용자 관점에서 작성. "오류 코드 E-404" → "페이지를 찾을 수 없습니다"
  • 2간결하고 명확하게 — 불필요한 단어 제거. 한 문장에 하나의 개념만
  • 3능동태 사용 — "파일이 삭제되었습니다" → "파일을 삭제했습니다"
  • 4구체적 행동 유도 — 다음에 무엇을 해야 하는지 명확하게 안내
  • 5일관된 용어 — 같은 기능·개념은 항상 같은 단어 사용
  • 6긍정적 표현 — "입력하지 않으면 안 됩니다" → "필수로 입력해 주세요"
Do
"인터넷 연결을 확인하고 다시 시도해 주세요."

"이름을 입력해 주세요."

"파일을 저장했습니다."
Don't
"네트워크 에러가 발생했습니다. (err: ETIMEDOUT)"

"이름 필드가 비어 있습니다."

"파일이 저장 완료되었습니다."
Brand Voice02

브랜드 보이스

KRDS 기반 서비스의 언어적 성격과 일관된 표현 방식입니다.

신뢰감 있고 전문적

정확한 정보와 명확한 안내로 신뢰를 구축합니다.

"본인 확인이 완료되었습니다."
"서류를 검토 중입니다. 영업일 기준 3일 내 안내드립니다."
친근하고 배려 있게

딱딱하지 않고, 사용자를 존중하는 말투를 사용합니다.

"잠깐이면 됩니다. 정보를 확인해 주세요."
"처음 사용하시나요? 단계별로 안내해 드릴게요."
명확하고 직접적

돌려 말하지 않고 핵심을 바로 전달합니다.

"신청 기간이 종료되었습니다. (마감일: 2025. 12. 31)"
"이미 신청한 내역이 있습니다."
해결 중심적

문제를 알리는 것에 그치지 않고 해결책을 함께 안내합니다.

"비밀번호를 잊으셨나요? 재설정 링크를 이메일로 보내드립니다."
"파일 크기가 초과되었습니다. 10MB 이하로 다시 업로드해 주세요."
Tone & Manner03

톤앤매너

상황에 따라 적절한 톤을 선택합니다. 기본 톤은 존중하는 격식체입니다.

상황톤예시주의
일반 안내친절·중립"확인해 주세요."—
오류 발생사실적·해결 중심"연결이 끊겼습니다. 다시 시도해 주세요."사과 남용 금지
경고·주의명확·긴박"삭제하면 복구할 수 없습니다."과장 금지
성공·완료긍정·간결"신청이 완료되었습니다."과도한 감탄 금지
온보딩안심·격려"어렵지 않아요. 함께 시작해 볼까요?"과도한 친근감 주의
데이터 삭제경고·신중"정말 삭제하시겠습니까? 복구가 불가능합니다."중립·객관적
경어 사용 원칙: "~해 주세요", "~하세요" 형태를 기본으로 합니다. "~하시기 바랍니다" 같은 과도한 경어 중복은 사용하지 않습니다.

Form & Input05

폼 & 입력

입력 폼은 최소한의 데이터만 수집하며, 사용자가 쉽게 이해하고 완료할 수 있도록 설계합니다.

플레이스홀더 원칙
플레이스홀더는 레이블이나 도움말의 대체 수단으로 사용하지 않습니다. 입력 형식 힌트(날짜, 자릿수 등)로만 활용합니다. (KRDS p.785)
Do — 형식 힌트만
예) 2025. 01. 01
예) 010-0000-0000
예) hong@email.com
Don't — 레이블 대체
이름을 입력하세요
생년월일
이메일 주소를 입력해 주세요
필수 / 선택 구분
  • ①필수 항목은 레이블 옆에 * (별표) 또는 [필수] 표시
  • ②선택 항목이 적은 경우 [선택]으로 표시하고 필수 표시 생략 가능
  • ③페이지 상단에 "* 표시 항목은 필수 입력 항목입니다" 안내 제공
예시
이름 *
이메일 주소 [필수]
연락처 [선택]
* 표시 항목은 필수 입력 항목입니다.
입력 제한 안내
최대 100자까지 입력할 수 있습니다.
영문·숫자 조합 8~20자로 입력해 주세요.
파일 크기는 10MB 이하만 업로드 가능합니다.
개인 식별 정보 원칙
  • 1이름: 성과 이름을 단일 필드에 입력 (성/이름 분리 금지)
  • 2생년월일: 날짜 선택기(Date Picker) 사용 금지. 텍스트 직접 입력 사용
  • 3성별: 기본값 지정 금지. '선택 안 함' 옵션 반드시 제공
  • 4잘못 입력한 값 자동 삭제 금지 — 사용자가 직접 수정하도록 유지
Error & Alert06

에러 & 알림

오류 메시지는 사용자가 문제를 빠르게 인지하고 해결할 수 있도록 작성합니다.

오류 메시지 4대 원칙 (KRDS p.829)
  • 1빠르게 인지 — 오류를 즉시 알아볼 수 있도록 시각적으로 명확하게 표현
  • 2분명하게 설명 — 무슨 오류인지, 왜 발생했는지 사용자가 이해할 수 있는 언어로
  • 3수정 방법 제공 — 어떻게 해결하면 되는지 구체적인 행동 안내
  • 4정중한 문체 — 사용자를 탓하지 않고 도움이 되는 방식으로 표현
폼 유효성 오류

Help Patterns07

도움말 패턴

KRDS는 6가지 도움 패턴을 정의합니다. 상황에 맞는 적절한 패턴을 선택해 사용합니다.

1. 안내 영역
페이지나 섹션 상단에 배치. 전체 맥락 안내에 활용.
개인정보는 서비스 제공 목적으로만 사용됩니다.
2. 플레이스홀더
입력 형식 힌트 전용. 레이블 대체 사용 금지.
예) 2025. 01. 01
3. 인라인 텍스트
입력 필드 아래 추가 설명. 항상 표시되어야 할 정보에 사용.
주민등록증에 기재된 이름을 입력해 주세요.
4. 툴팁
물음표 아이콘 호버 시 표시. 보조 정보에 활용.
공동인증서 또는 금융인증서로 본인을 확인합니다.
5. 맥락적 도움말
사용자 행동에 반응하여 표시. 실시간 유효성 안내.
사용 가능한 비밀번호입니다.
6. 코치마크
신규 기능이나 첫 방문 시 UI 요소를 가리키며 안내.
여기를 눌러 즐겨찾기에 추가할 수 있습니다.
Navigation08

내비게이션

메뉴 레이블과 경로 안내 문구는 사용자가 현재 위치와 이동 방향을 명확히 파악할 수 있도록 작성합니다.

메뉴 레이블 원칙
  • 1명사형 또는 명사구 사용 (동사형 지양)
  • 2약어 지양. 불가피한 경우 처음 사용 시 풀어서 표기
  • 31~3단어로 간결하게 유지
브레드크럼 & 단계 표시
홈 > 민원 신청 > 신청서 작성
1단계 정보 입력 → 2단계 서류 제출 → 3단계 신청 완료
총 3단계 중 2단계
Onboarding09

온보딩

처음 사용하는 사용자가 서비스를 쉽게 시작할 수 있도록 안심시키고 단계별로 안내합니다.

시작 화면 / 환영 문구
안녕하세요! 처음 오셨군요. 시작하는 방법을 알려드릴게요.
3분이면 완료됩니다. 필요한 서류를 미리 준비해 주세요.
주민등록증 또는 운전면허증이 필요합니다.
단계별 진행 안내
거의 다 왔습니다. 마지막으로 서류를 확인해 주세요.
이 단계는 건너뛸 수 있습니다. 나중에 설정에서 변경 가능합니다.
입력한 정보를 확인해 주세요. 제출 후에는 일부 항목을 수정할 수 없습니다.
Empty State10

빈 상태

내용이 없는 상태에서 다음 행동을 명확하게 유도합니다. 구조: [이유] + [행동 유도]

신청 내역이 없습니다. 지금 바로 신청해 보세요.
검색 결과가 없습니다. 다른 검색어로 찾아보세요.
저장된 즐겨찾기가 없습니다. 자주 사용하는 서비스를 즐겨찾기에 추가해 보세요.
알림이 없습니다.
아직 작성한 글이 없습니다. 첫 번째 글을 작성해 보세요.

Accessibility13

접근성 (KWCAG 2.2 / WCAG 2.1)

모든 사용자가 서비스를 이용할 수 있도록 접근성 기준을 준수합니다.

스크린리더 aria-label 원칙
  • 1아이콘 전용 버튼에는 반드시 aria-label 제공
  • 2이미지에는 의미 있는 alt 텍스트 제공 (장식 이미지는 alt="")
  • 3오류 발생 시 해당 입력 요소로 초점 자동 이동
  • 4색상 대비 최소 4.5:1 (텍스트), 3:1 (대형 텍스트)

Internationalization14

국제화 & 다국어

다국어 서비스를 위한 UX 라이팅 고려사항입니다.

번역 확장 고려 원칙
  • 1한국어 대비 영어 1.5~2배, 독일어 2~3배 길어질 수 있음 — 레이아웃에 여유 공간 확보
  • 2줄임말·속어·관용표현 지양 (번역 시 의미 손실 가능)
  • 3날짜 형식: YYYY. MM. DD (한국) / MM/DD/YYYY (미국) — 변수로 분리
  • 4숫자 단위, 화폐 기호, 전화번호 형식을 하드코딩하지 않음

Checklist15

UX Writing 체크리스트
