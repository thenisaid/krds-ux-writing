원칙

왜 이렇게 써야 할까

글쓰기 기술은 나중입니다. 먼저 '누구를 위한 글인가'를 알아야 합니다. 공공 서비스의 글은 행정 정확성보다 주권자의 즉각적인 이해가 우선입니다. 3가지 라이팅 원칙은 그 철학에서 출발합니다.

원칙 살펴보기 →

기초 원칙

01
기초
번역 금지 (No Direct Translation)

행정 용어를 그대로 디지털 인터페이스에 옮기지 마세요. 법률문은 법원에서, 서비스 문구는 시민에게. 두 언어를 구분하는 것이 UX 라이팅의 시작입니다.

→
02
기초
핵심 정보 먼저 (Core Info First)

사용자는 전체를 읽지 않습니다. 역피라미드 구조로 가장 중요한 정보를 첫 줄에 배치하세요. 부연 설명은 그 다음입니다.

→
03
기초
안전망 원칙 (Safety Net)

사용자가 막혔을 때, 혼자 해결할 수 있도록 항상 다음 단계를 안내하세요. 오류 메시지, 막힘 화면, 지연 안내 모두 안전망이 필요합니다.

→

보이스 · 톤

보이스
심화
보이스·톤 가이드

공공 서비스의 목소리는 권위적이지 않으면서도 신뢰감이 있어야 합니다. 상황별 톤 조절 방법과 금지 표현 목록을 제공합니다.

→
격식
심화
공식성 스펙트럼

경어체와 평어체 선택 기준, 서비스 성격에 따른 격식 수준 가이드. "합니다/해요/해라" 체를 언제 어떻게 사용할지 기준을 제시합니다.

→

컴포넌트별 라이팅

BTN
실전
버튼 라이팅

동사로 시작하고, 결과를 명시하세요. "확인"이 아닌 "신청하기", "저장하기". KRDS 버튼 컴포넌트별 권장 레이블 패턴을 제공합니다.

→
폼
실전
폼 · 입력 라이팅

레이블, 플레이스홀더, 도움말, 인라인 오류의 4가지 텍스트 영역. 각각의 역할과 권장 패턴, 반면교사 사례를 정리했습니다.

→
오류
실전
오류 메시지 라이팅

"오류가 발생했습니다"는 도움이 안 됩니다. 무슨 일이 생겼는지, 왜 생겼는지, 어떻게 해야 하는지 — 세 가지를 담은 오류 메시지 공식.

→
피드백
실전
시스템 피드백 문구

성공, 로딩, 빈 상태, 시간 초과. 사용자가 시스템 상태를 이해할 수 있도록 돕는 피드백 메시지 패턴 가이드.

→
온보딩
실전
온보딩 라이팅

처음 사용자가 서비스를 이해하고 시작할 수 있도록 돕는 온보딩 문구. 환영 메시지, 튜토리얼, 빈 화면 텍스트 패턴.

→
모바일
심화
모바일 UX 라이팅

화면이 작을수록 언어는 더 간결해야 합니다. 모바일 공공앱에 최적화된 문구 길이 기준, 알림 문구, 터치 액션 레이블 가이드.
