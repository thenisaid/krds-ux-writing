## 2. 페르소나 및 사용자 여정 요약

### 2.1 페르소나 정의

| 페르소나 | 이름 (직급) | 목표 | 주요 니즈 | 기술 숙련도 |
|---------|------------|------|----------|-----------|
| **UX 라이터** | 김민지 (3년차) | 정부 서비스 UI 문구 작성 시 검증 기준 확보, 팀 내 용어·톤앤매너 가이드 정립 | 즉시 적용 가능한 예시 문구, 팀 공유 가능한 링크, 자가 점검 체크리스트 | 높음 (⌘K 단축키, 복사 버튼 적극 활용) |
| **프로덕트 디자이너** | 이준호 (5년차) | Figma 컴포넌트 레이블·플레이스홀더 기준 확보, 디자인 리뷰 시 라이터와 협업 기준 마련 | 버튼/폼/에러 섹션 Do/Don't 예시, 디자이너 전용 가이드, 빠른 내비게이션 | 높음 (드롭다운 메뉴로 원하는 섹션 직접 접근) |
| **프론트엔드 개발자** | 박서연 (2년차) | aria-label, 접근성 구현 기준 확인, 에러 메시지 문구 코드 적용 | aria-label 작성 가이드, 접근성 & 국제화 섹션, 복사 가능한 코드 수준 예시 | 중간 (검색보다 스크롤 탐색 선호) |

### 2.2 여정별 핵심 인사이트

#### UX 라이터 (김민지)
**감정 곡선**: 진입 😊 → 탭 탐색 😊 → 카드 확인 😊 → 가이드 읽기 😊 → 복사 사용 😄 → 체크리스트 😊(약간 불편: 상태 초기화) → 재방문 😊

**핵심 발견**
- 복사 버튼이 이 페르소나의 핵심 인터랙션 포인트 (빈도 높음)
- Do/Don't 비교 형식이 실무 적용에 직접적으로 유효
- 체크리스트 상태 유지 부재가 실무 워크플로 단절 초래
- 역할 탭 전환 오류는 팀 협업 시나리오에서 경험 저하 요인

**주요 페인포인트**
- 복사 후 Figma 이동 → 다시 돌아올 때 어디 있었는지 위치 손실
- 체크 상태가 새로고침 시 초기화됨 (localStorage 미사용 추정)
- 검색 모달에 실제 검색 결과가 나오는지 확인 필요 (UI만 있고 기능 미구현 가능성)

#### 프로덕트 디자이너 (이준호)
**감정 곡선**: 진입 😊 → GNB 탐색 😊 → 디자이너 탭 😟(버그 실망) → 버튼 섹션 😊 → 폼 섹션 😐(약간 불편) → 에러 섹션 😊 → 접근성 😊(협업 가치)

**핵심 발견**
- **디자이너 탭 JavaScript 오류**가 이 페르소나에게 가장 직접적인 충격 지점
- 버튼/폼 섹션의 카테고리 분류가 Figma 컴포넌트 구조와 잘 맞아 협업 가치 높음
- 페이지 단일 스크롤 구조로 인해 긴 콘텐츠 탐색 시 피로도 증가
- 팀 협업 도구로서의 잠재력 높지만 디자이너 전용 콘텐츠 부재

**주요 페인포인트**
- **탭 전환이 작동하지 않음 (JavaScript TypeError)** — 핵심 페인포인트
- 같은 섹션 내 항목이 많아 스크롤이 길어짐. 특정 레이블 검색 불가
- 원칙과 예시가 분리되어 있어 왔다 갔다 해야 함
- 접근성 섹션이 페이지 최하단에 위치 — 중요도 대비 접근성 낮음

#### 프론트엔드 개발자 (박서연)
**감정 곡선**: 진입 😐 → 개발자 탭 😟(실망) → 스크롤 탐색 😐 → 에러 섹션 😊 → 접근성 섹션 😊 → 검색 시도 😐(불확실) → 국제화 섹션 😐(아쉬움)

**핵심 발견**
- 개발자 페르소나는 가이드라인 페이지보다 구체적인 코드/구현 레퍼런스를 원함
- 복사 기능이 유용하지만 개발자 친화적 형식(JSON, code snippet)이 부재
- 접근성 가이드의 내용은 충실하나 "구현 방법"보다 "원칙 설명"에 치우쳐 있음
- 검색 기능의 실제 동작 여부가 불확실해 재방문 효율성 저하 우려

**주요 페인포인트**
- 탭 전환 오류 가능성 (디자이너 탭과 동일 JavaScript 이슈)
- 복사한 문구가 일반 텍스트 — JSON key-value 형태나 i18n 포맷으로 복사 가능하면 더 좋겠음
- 가이드는 있지만 실제 HTML 코드 예시(aria-label="메뉴 열기" 속성이 들어간 button 태그)가 없음
- 국제화 내용이 개념 수준에만 머물고 실제 구현 가이드(react-i18next, 한국어 날짜 포맷 코드)가 없음

### 2.3 크로스-페르소나 공통 페인포인트

모든 페르소나에 걸친 공통 문제 5가지:

1. **역할 탭 전환 버그**: 디자이너/개발자 탭 클릭 시 `TypeError: Cannot read properties of undefined` 발생. 세 페르소나 모두 역할별 맞춤 콘텐츠 기대 — **가장 즉각적으로 수정이 필요한 버그**

2. **스크롤 내비게이션 피로**: 단일 페이지 스크롤 구조(13,377px)에서 9개 섹션 탐색 시 현재 위치 파악이 어려움. 스티키 사이드바 목차 또는 진행 표시자 부재

3. **검색 기능 완결성 불확실**: 검색 모달 UI는 존재하나 실제 검색 결과 표시 기능의 구현 여부 불명확. 재방문 사용자의 핵심 탐색 도구

4. **체크리스트 상태 비영속성**: 페이지 새로고침 시 체크 상태 초기화. 실무에서 세션 간 진행 상황 유지 불가

5. **앵커 이동 오프셋**: GNB 높이로 인해 앵커 이동 시 섹션 제목이 헤더에 가려질 가능성 (`scroll-margin-top` 미적용 추정)

### 3.2 치명적/중요 이슈 상세

#### H5-1: 역할별 탭 전환 JavaScript 오류 [심각도 3/4 — 치명적]
**위치**: 역할별 탭 (디자이너/개발자 탭)
**증상**: "디자이너", "개발자" 탭 클릭 시 콘텐츠가 전환되지 않음. `switchPersonaTab` 함수에서 `TypeError: Cannot read properties of undefined (reading 'classList')` 발생
**영향**: 디자이너/개발자 페르소나의 핵심 여정 단절. 버튼 클릭은 가능하나 실제 동작이 없어 사용자를 오도함
**개선**: `switchPersonaTab` 함수의 undefined 참조 오류 즉시 수정 (패널 ID 참조 로직 검토)

#### H3-1: 검색 모달 ESC 키 동작 불일치 [심각도 2/4 — 중요]
**위치**: 검색 모달
**증상**: ESC 키를 한 번 눌렀을 때 닫히지 않고 두 번 눌러야 닫힘. 첫 번째 ESC는 효과 없이 무시됨
**영향**: 표준 모달 동작(ESC=즉시 닫기) 위반. 키보드 사용자 경험 저하
**개선**: ESC 첫 입력에 즉시 검색 모달 닫힘 처리

#### H5-2: "품질 기준" 메뉴 연결 불명확 [심각도 2/4 — 중요]
**위치**: GNB
**증상**: "품질 기준" 드롭다운 메뉴 클릭 시 내부 링크가 어디로 연결되는지 불분명. 클릭 후 상태 변화가 없을 경우 사용자가 사이트 오류로 오인 가능
**영향**: 사용자 기대와 실제 내용 불일치
**개선**: "품질 기준" 메뉴 연결 콘텐츠 확보 또는 메뉴 제거

#### H6-1: 긴 단일 페이지 현재 섹션 위치 파악 어려움 [심각도 2/4 — 중요]
**위치**: 전체 페이지
**증상**: 페이지가 매우 긴 단일 HTML (스크롤 높이 13,377px)이지만 현재 섹션 위치를 나타내는 고정형 사이드바 또는 목차가 없음
**영향**: 스크롤 중 어느 섹션에 있는지 파악하기 어려움. 특히 디자이너 페르소나의 탐색 피로도 증가
**개선**: 스크롤 연동 사이드 목차(sticky TOC) 또는 "현재 위치: 섹션명" 표시 추가

#### H6-2: GNB 현재 섹션 자동 활성화 부재 [심각도 2/4 — 중요]
**위치**: GNB
**증상**: 스크롤하여 특정 섹션을 볼 때 GNB의 해당 메뉴 항목이 활성화(highlighted)되지 않음. 현재 읽고 있는 섹션이 어느 메뉴에 해당하는지 기억에 의존해야 함
**영향**: 긴 문서 탐색 시 현재 위치 인식 어려움
**개선**: Intersection Observer로 현재 뷰포트의 섹션에 해당하는 GNB 항목 자동 활성화

#### H4-1: 탭 패널 aria-label 부재 [심각도 2/4 — 중요]
**위치**: 역할별 탭 패널
**증상**: 탭 패널(`aria-label`)이 없음. `aria-label="UX 라이터 탭 패널"` 형태의 레이블이 없어 스크린리더 사용자가 어느 탭 내용인지 파악 불가
**영향**: 접근성 저하
**개선**: 탭 패널에 `aria-label` 속성 추가

#### H4-2: GNB-콘텐츠 구조 불일치 [심각도 2/4 — 중요]
**위치**: GNB 드롭다운
**증상**: 섹션이 01~09까지 있지만 GNB 드롭다운에는 일부 섹션만 포함됨 (버튼 & CTA, 폼 & 입력이 "가이드라인" 메뉴에 없고 "패턴 라이브러리"에 일부 있음). 사이트 구조와 메뉴 구조 불일치
**영향**: 사용자가 원하는 섹션을 메뉴에서 찾지 못함
**개선**: GNB 메뉴 구조를 콘텐츠 섹션 구조와 정렬

#### H9-1: 탭 전환 오류 진단 불가 [심각도 2/4 — 중요]
**위치**: 역할별 탭
**증상**: 디자이너/개발자 탭 클릭 시 콘텐츠가 변하지 않아 사용자가 오류인지 인지하지 못함. 오류 메시지도 없고 시각적 피드백도 없어 문제 진단 불가
**영향**: 사용자가 시스템 오류를 인지하지 못하고 반복 클릭 또는 이탈
**개선**: 탭 전환 오류 수정이 근본 해결책. 임시 방편으로 탭 전환 실패 시 "일시적인 오류가 발생했습니다" 토스트 알림 추가

### 3.3 강점 (잘된 점)

#### 즉각적 피드백 시스템
- 복사 버튼 클릭 시 "✓ 복사됨"으로 즉각 피드백 제공 (버튼 텍스트 및 색상 변화)
- 체크리스트 항목 클릭 시 체크 표시 + 취소선으로 완료 상태 시각적 표현
- 다크/라이트 모드 전환 즉각 반응

#### 접근성 기본 준수
- 모든 버튼에 aria-label 또는 텍스트 레이블 존재 (55개 버튼 중 레이블 누락 0개)
- 스크롤 진행바(scroll-progress)가 상단 고정 위치에 표시되어 문서 진행 위치 파악 가능
- 모든 레이블이 한국어로 작성되어 대상 사용자(한국 공공서비스 담당자)와 일치

#### 키보드 지원 및 효율성
- ⌘K 키보드 단축키로 검색 빠르게 열기 가능 (실제 동작 확인)
- ESC 키로 검색 모달 닫기 가능 (두 번 입력 필요하지만 동작)
- GNB에서 앵커 링크로 원하는 섹션으로 바로 이동 가능
- 복사 버튼으로 예시 문구를 클립보드에 즉시 복사 가능 (실무 효율성 높음)

#### 콘텐츠 품질 및 실무 활용성
- 가이드라인 자체가 UX Writing 원칙을 충실히 따름 (사용자 중심 언어, 해결책 포함 오류 메시지 등)
- "✓ Do / ✗ Don't" 패턴으로 직관적인 옳고 그름 구분
- Do/Don't 영역이 명확히 구분되어 시각적 노이즈 최소화
- 아이콘과 레이블이 함께 사용되어 의미 전달 강화

#### 시각 디자인
- 다크/라이트 모드 모두 깔끔하고 일관된 색상 팔레트 사용
- 히어로 섹션의 배경 애니메이션이 과하지 않고 브랜드 감성 강화
- 섹션 간 충분한 여백으로 가독성 확보
- 색상 체계(파란색 Primary, 빨간색 위험)가 일관적으로 사용됨

## 결론

KRDS UX Writing 페이지는 **콘텐츠 품질 및 실무 활용성** 측면에서 우수한 강점을 보이며, 복사 버튼, Do/Don't 예시, 체크리스트 등이 실제 UX Writing 실무에 즉시 적용 가능한 도구로 기능합니다.

그러나 **역할별 탭 전환 JavaScript 오류 (H5-1, 심각도 3/4)**로 인해 디자이너 및 개발자 페르소나의 핵심 여정이 단절되어 있으며, 이는 즉시 수정이 필요한 치명적 이슈입니다. 또한 긴 단일 페이지 구조(13,377px)에서 스크롤 위치 파악 도구가 부재하여 탐색 효율성이 저하되고 있습니다.

Phase 1 (즉시 수정) 및 Phase 2 (단기 개선) 작업을 완료하면 전체 사용성 점수가 30-40% 향상될 것으로 예상되며, 특히 역할별 맞춤 콘텐츠 소비율과 재방문 효율성이 크게 개선될 것입니다.

개선 작업은 기존의 우수한 콘텐츠를 유지하면서 **도구로서의 접근성 및 효율성**을 강화하는 데 집중하여, KRDS UX Writing 페이지가 공공 서비스 UX Writing의 표준 레퍼런스로 자리잡을 수 있도록 합니다.
