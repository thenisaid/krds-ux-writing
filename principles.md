## 서문

**이 가이드라인의 목표는 단 하나입니다.**
**누구나 포기하지 않고 끝까지 쓸 수 있게 만드는 것.**

공공 서비스의 글은 멋있거나 유려할 필요가 없습니다. 국민이 헷갈리지 않고, 제때 세금을 내고, 필요한 지원금을 무사히 받게 돕는 **'친절한 표지판'** 이어야 합니다.

우리는 그동안 글을 쓸 때 '명확하게, 간결하게, 유용하게, 일관되게' 쓰라는 말을 수없이 들어왔습니다. 하지만 이 모호한 형용사들은 현장에서 아무런 도움이 되지 않았습니다. 그래서 KRDS는 이 4가지 기준을 실무자가 당장 모니터 앞에서 쓸 수 있는 **'구체적인 행동 규칙'** 으로 바꿨습니다.

| 기존 기준 | KRDS 행동 규칙 |
|-----------|---------------|
| 명확하게 | 무번역의 원칙 — 14세 중학생이 사전 없이 이해하는 일상어 |
| 간결하게 | 정보 핵심화의 원칙 — 결론과 다음 버튼을 맨 위에 |
| 유용하게 | 심리적 안전망의 원칙 — 실수했을 때 원인+해결책 함께 |
| 일관되게 | 세 원칙을 예외 없이 지킬 때 쌓이는 신뢰 |

어디를 가든 쉬운 말로 안내받고 실수해도 괜찮은 경험이 쌓일 때, 비로소 시스템과 국민 사이에 **"정부 서비스는 헷갈리지 않는다"** 는 보이지 않는 신뢰의 선이 연결됩니다.

## 제0장. 적용 범위 (Scope)

> 이 가이드라인이 어디에, 얼마나, 어떻게 적용되는지 먼저 확인하세요.

---

### 0.1 표준형 — 중앙행정기관 전면 적용

KRDS는 **표준형(Standard)**과 **확장형(Adaptive)** 두 트랙으로 운영됩니다.

**표준형** 적용 기관은 정부 상징 로고를 사용하는 중앙행정기관입니다. 이 가이드라인의 **모든 원칙을 예외 없이 적용**합니다.

| 항목 | 적용 기준 |
|------|----------|
| 무번역의 원칙 | 필수 — 대체어 사전 전면 준수 |
| 정보핵심화의 원칙 | 필수 — 결론 우선 구조, 글자 수 기준 준수 |
| 심리적안전망의 원칙 | 필수 — 에러 3단 구조, 성공 메시지 패턴 준수 |
| 보이스·톤 | KRDS 기본 보이스 적용 (문어체·격식체) |
| 컴포넌트 레이블 | KRDS 토큰 의미에 맞는 문구 필수 |

**레이블·오류 문구·안내문**을 작성할 때, 해당 UI 컴포넌트의 KRDS 디자인 토큰 역할(role)과 의미(semantic)를 먼저 확인한 뒤 문구를 선택합니다. 토큰 역할이 `primary-action`이면 레이블도 주된 행동을 표현하는 동사여야 합니다.

---

### 0.2 확장형 — 공공기관·운영 서비스 적용 기준

**확장형** 적용 기관은 독자 로고를 사용하는 공공기관, 산하기관, 운영 서비스(예: 온통청년, 정부24 파생 서비스)입니다.

| 항목 | 적용 기준 |
|------|----------|
| 무번역의 원칙 | **필수** — 행정어 배제 기준은 동일 적용 |
| 정보핵심화의 원칙 | **필수** — 역피라미드 구조 준수 |
| 심리적안전망의 원칙 | **필수** — 에러·성공 메시지 원칙 준수 |
| 보이스·톤 | **기관 재정의 허용** — 단, KRDS 기본 보이스 대비 이탈 근거 문서화 필요 |
| 컴포넌트 레이블 | KRDS 토큰 기반 준수 + 기관 특수 용어 사전 추가 허용 |

**확장형은 무엇을 바꿀 수 있는가:**
- ✅ 종결 어미 (예: `~합니다` → `~해요`)
- ✅ 대명사 (예: `고객님` → `회원님` → 기관 맞춤 호칭)
- ✅ 기관 특수 용어 사전 추가 (단, 일반어 우선 원칙 위반 불가)
- ❌ 무번역 원칙 자체를 완화하거나 예외 허용 불가
- ❌ 에러 메시지에서 원인·해결 정보 제거 불가

---

### 0.3 KRDS 디자인 토큰 구조와 UX Writing 연계

UX Writing 원칙은 KRDS 디자인 토큰 체계와 동일한 5계층 구조 위에서 작동합니다.

```
L1 Primitive   색상·타이포·간격·아이콘 원시값
L2 Semantic    역할 기반 토큰 (--color-primary-50, --text)
L3 Component   컴포넌트 단위 토큰 (--krds-color-button-fill-primary-label-default)
L4 Pattern     반복 과업 흐름 (검색·로그인·신청)
L5 Service     사용자 여정 전체
```

**UX Writing 적용 레이어:**

| 계층 | UX Writing 역할 |
|------|---------------|
| L2 Semantic | 토큰 네이밍 원칙 — 역할 기반, 약어 금지, 시각 속성 배제 |
| L3 Component | 컴포넌트별 레이블·플레이스홀더·상태 문구 기준 (6장) |
| L4 Pattern | 패턴 흐름별 안내문 원칙 (3장·4장) |
| L5 Service | 서비스 전반 톤앤매너 (1.4, 5장) |

**컴포넌트 상태와 UX Writing 대응:**

KRDS는 11개 UI 상태 토큰을 정의합니다. 각 상태에 대응하는 문구 기준이 있어야 합니다.

| 상태 토큰 | UX Writing 기준 |
|---------|---------------|
| `default` | 기능 기반 레이블 (동사형 또는 명사형 예외 적용) |
| `hover` | 별도 문구 없음 (시각 피드백으로 처리) |
| `focused` | 입력 필드: 플레이스홀더 표시 (레이블 대체 금지) |
| `disabled` | 사용 불가 이유를 근접 설명 텍스트로 명시 |
| `error` | 원인 + 결과 + 해결 3단 구조 (4.1절 준수) |
| `completed` | 완료 사실 + 다음 행동 안내 (4.2절 준수) |
| `selected` / `unselected` | 선택 상태 변화는 레이블이 아닌 아이콘+색상으로 처리 |
| `indeterminate` | 부분 선택 상태 — "일부 선택됨" 등 상태 명시 필요 |
| `active` | 현재 위치 표시 — 탐색 컴포넌트(탭·메뉴)에서 사용 |

**Depth(Elevation)와 문구 위계:**

화면의 Depth가 높을수록 해당 레이어의 문구는 더 핵심적인 행동을 요구합니다.

| Depth | 컴포넌트 예 | 문구 원칙 |
|-------|-----------|---------|
| Depth 0 (Background) | 페이지 본문 | 설명·안내 중심 |
| Depth 1 (Surface) | 카드·패널 | 결론 우선, 핵심 정보 |
| Depth 4 (Modal/Overlay) | 모달·다이얼로그 | 단문 헤더 + 단일 CTA |
| Depth 5 (Toast/Snackbar) | 토스트·스낵바 | 1문장, 버튼 최대 1개 |

> 모달과 다이얼로그는 가장 높은 집중도를 요구하는 레이어입니다.
> 헤더는 목적을 명시하고, 버튼은 행동을 반영하며, **"모달은 최후의 수단이다"** (KRDS 공식 원칙)를 반드시 기억하세요.

## 제1장. 파운데이션 (Foundation)

---

### 1.1 비전: 시민이 읽는 글, 국가가 쓰는 말

공공 서비스의 언어는 국가가 시민에게 보내는 가장 직접적인 신호입니다.

'수급', '교부', '귀하'처럼 어려운 말은 시민을 배제하는 신호입니다. '혜택', '발급', '고객님'처럼 쉬운 말은 국가가 시민의 편임을 증명하는 신호입니다.

이 가이드라인은 그 신호를 바꾸는 작업입니다. 어디서나 쉽고, 어디서나 일관되고, 어디서나 안심할 수 있는 공공 언어 — **시민이 읽는 글, 국가가 쓰는 말** 을 만드는 것이 우리의 비전입니다.

---

### 1.2 주권자: 이 가이드를 쓰는 사람 / 읽는 사람

**이 가이드를 읽는 사람** — UX 라이터, 콘텐츠 기획자, 디자이너, 개발자, 정부 관계자. 화면에 들어갈 문장을 최종 결정하는 모든 사람.

**이 가이드가 지키려는 사람** — 공공 서비스를 이용하는 시민. 이들은 행정 절차를 배우러 온 학생이 아닙니다. 낯선 시스템 앞에서 실수하여 불이익을 받을까 봐 초조해하며, 자신의 목적을 가장 효율적으로 달성하고 일상으로 돌아가길 원하는 **'주권자'** 입니다.

실무자는 글을 쓰기 전, 주권자가 가진 3가지 정당한 기대와 상황적 특성을 반드시 이해해야 합니다.

#### ① 정보의 효율성에 대한 기대 — 목적 지향적 스캐닝

주권자는 화면을 꼼꼼히 정독하지 않고, 자신이 달성해야 할 목적(버튼, 핵심 결과)만 빠르게 훑어봅니다. 가장 중요한 결론과 다음 행동 지시를 화면 최상단과 버튼에 배치해야 합니다.

#### ② 보편적 이해에 대한 권리 — 14세 기준 무장벽 언어

공공의 언어는 특정 계층의 전유물인 '한자어'나 '행정 관습어'의 장벽을 두어서는 안 됩니다. 누구도 정보에서 배제하지 않도록 문장의 수준을 중학교 2학년(14세) 독해력에 맞춥니다.

#### ③ 심리적 안전망에 대한 요구 — 실수 복구의 권리

공공 서비스에서의 클릭은 무겁습니다. 사용자의 실수를 시스템이 탓하지 않아야 하며, 오류가 발생했을 때 '실패 통보'로 끝내지 않고 '어떻게 궤도를 수정할 수 있는지' 명확한 대안을 제시해야 합니다.

---

### 1.3 3대 원칙 개요: 주권자의 시간을 아끼는 실행 기준

공공 서비스의 글은 주권자가 단어의 뜻을 유추하느라 멈춰 서지 않고, 안심하며 다음 버튼을 누를 수 있도록 돕는 실용적인 표지판이어야 합니다.

---

#### ⚖️ 원칙 1. 무번역의 원칙 (No-Translation)

공급자의 행정적 정확성보다 주권자의 즉각적인 이해가 우선입니다.
뇌가 일상어로 다시 '번역'해야 하는 단어는 철저히 배제합니다.

**행정 관습어 폐기**
'명일', '기표', '내방' 등 관공서 은어는 '내일', '등록', '방문' 등 보편적인 일상어로 1:1 치환합니다.

| 쓰지 마세요 | 대신 쓰세요 |
|------------|------------|
| 명일까지 제출 | 내일까지 제출 |
| 직접 내방하여 | 직접 방문하여 |
| 기표 완료 | 등록 완료 |
| 상기 사항 | 위 내용 |
| 동 기간 내 | 같은 기간 안에 |

**이중 부정 금지**
뇌의 인지 부하를 높이는 이중 부정 대신, 한 번에 읽히는 긍정문으로 씁니다.

| 이중 부정 | 긍정문 |
|----------|--------|
| 미비하지 않으면 신청 가능합니다 | 모두 준비되면 신청할 수 있습니다 |
| 불가능하지 않은 경우 | 가능한 경우 |

---

#### ⚖️ 원칙 2. 정보 핵심화의 원칙 (Essentialism)

무작정 글자 수를 줄이는 것이 아니라, 주권자가 핵심에 도달하는 생각의 거리를 물리적으로 줄여주는 **'노이즈 제거'** 과정입니다.

**결론 우선 (Front-loading)**
주권자가 이 화면에서 취해야 할 핵심 행동이나 최종 결론을 맨 앞에 배치합니다.

> ❌ 귀하의 소득 및 재산 현황을 검토한 결과, 지원 대상에 해당되지 않음을 알려드립니다.
> ✅ 지원 대상이 아닙니다. 소득 기준을 초과했습니다.

**당연한 말 생략**
무의미한 인사말이나 시스템이 당연히 해야 할 일에 대한 수식어를 생략하고 핵심 정보만 남깁니다.

> ❌ 소중한 개인정보 보호를 위해 안전하게 처리됩니다.
> ✅ (삭제 — 당연한 사항)

**시각적 리듬과 청킹 (Chunking)**
한 문장에 담는 의미의 마디(청크)는 3개를 넘지 않도록 합니다. 의미 단위로 줄을 바꿔 스캐닝을 돕습니다.

---

#### ⚖️ 원칙 3. 심리적 안전망의 원칙 (Psychological Safety)

실수해도 언제든 궤도를 수정할 수 있다는 확신을 주어 비가역적 공포를 없앱니다.

**100% 예측 가능한 버튼**
'확인/취소' 같은 모호한 단어 대신, 누른 뒤 일어날 일을 명확히 적습니다.

| 모호한 버튼 | 예측 가능한 버튼 |
|------------|----------------|
| 확인 | 신청서 제출하기 |
| 취소 | 작성 취소하고 나가기 |
| 다음 | 본인 인증으로 이동 |
| 완료 | 신청 완료 — 접수번호 확인하기 |

**비난 없는 오류 복구**
시스템 오류 코드만 내뱉지 않고, '무엇이 문제인지 + 어떻게 고치는지'를 함께 제시합니다.

> ❌ ERROR 4023: 인증 실패
> ✅ 인증 번호가 틀렸습니다. 문자로 받은 6자리 숫자를 다시 입력해 주세요.

**비가역 행동 앞의 제동**
최종 제출 등 되돌릴 수 없는 행동 직전에는 반드시 명확한 경고로 흐름을 끊고 시야를 집중시킵니다.

> 제출하면 수정할 수 없습니다. 내용을 다시 확인해 주세요.

### 1.4 보이스·톤 기준: 주권자를 대하는 정부의 일관된 인격

공공 서비스의 인격은 상황에 따라 오락가락해서는 안 됩니다. 흔들리지 않는 하나의 **보이스(Voice)** 를 유지하되, 주권자의 감정 곡선에 맞춰 **말투(Tone)** 를 4단계로 유연하게 변주합니다.

---

#### 🏛️ 우리의 보이스 (Voice): '권위를 뺀 든든한 조력자'

우리는 가르치거나 통제하지 않으며, 다음 3가지 성격을 항상 유지합니다.

1. **객관적이고 투명한 (Transparent):** 제한 사항을 숨기지 않고 명확히 밝힙니다.
2. **단호하지만 책임을 지는 (Accountable):** 안 되는 것은 선을 긋되, 대안을 책임지고 안내합니다.
3. **담백하게 정중한 (Respectful):** 불필요한 감정 표현(이모지)이나 과도한 극존칭을 빼고 존중의 의미만 남깁니다.

---

#### 🎛️ 상황별 톤 매트릭스 (Tone Matrix)

| 상태 | 톤 | 주권자 감정 | 원칙 |
|------|----|------------|------|
| 🟢 탐색과 안내 | 길잡이 | 안정 | 군더더기 없이 객관적·명확 |
| 🟡 대기와 로딩 | 중계자 | 초조·의심 | 현재 상황과 예상 시간 투명 공개 |
| 🔴 오류와 실패 | 해결사 | 당황·분노 | 원인+해결책만 건조하게, 이모지 금지 |
| 🔵 완료와 성공 | 마침표 | 안도 | 과업 완료 확인 + 다음 일정 안내 |

**🟢 톤 1. 탐색과 안내 (안정 상태)**
목적을 향해 가는 주권자를 위해 군더더기 없이 객관적이고 명확한 길잡이 역할을 합니다.
> *어떤 서비스가 필요하신가요? 아래에서 선택해 주세요.*

**🟡 톤 2. 대기와 로딩 (초조/의심 상태)**
막연히 기다리게 하지 않고, 시스템이 현재 무엇을 하고 있는지, 얼마나 걸리는지 투명하게 중계하여 안심시킵니다.
> *국세청에서 소득 정보를 불러오고 있습니다. (약 10초 소요)*

**🔴 톤 3. 오류와 실패 (당황/분노 상태)**
감정을 철저히 배제하고 가장 건조한 톤으로 '원인'과 '해결책'만 단호하게 짚어줍니다. 조롱으로 느껴질 수 있는 이모지나 느낌표는 절대 금지합니다.
> *인증 번호가 일치하지 않습니다. 문자로 받은 6자리 숫자를 다시 입력해 주세요.*

**🔵 톤 4. 완료와 성공 (안도 상태)**
과도하게 텐션을 높여 축하하지 않고, 과업이 확실히 끝났다는 마침표를 찍어주며 다음 일정을 안내합니다.
> *지원금 신청이 끝났습니다. 심사 결과는 3일 뒤 카카오톡으로 알려드릴게요.*

### 2.1 부록. 약관·법적 고지 예외 처리

이용약관·개인정보처리방침은 법률 원문을 그대로 인용해야 하는 경우가 많아, 행정어 기준을 일괄 적용할 수 없습니다.

**권장 처리 방식**:
1. 법률 원문을 유지하되, **요약 버전을 먼저 제시** (결론 우선 원칙 적용)
2. "이 내용을 쉽게 말하면:" 섹션을 추가해 핵심만 일상어로 설명
3. 법률 조항 번호(제15조 제1항)는 생략하거나 괄호 처리

**예시**:
```
✅ 우리가 수집하는 정보: 이름, 전화번호, 이메일
   이 정보는 서비스 가입과 본인 확인에만 씁니다.
   (법적 근거: 개인정보 보호법 제15조 제1항 제2호)
```

### 2.2 폼 레이블·플레이스홀더 작성 기준

폼은 시민이 공공 서비스와 직접 소통하는 접점입니다. 레이블과 플레이스홀더는 시민이 "무엇을, 어떻게, 왜 입력해야 하는지"를 인지 부하 없이 파악하게 돕는 이정표입니다.

---

#### 레이블 작성 원칙

**원칙 1 — 명사형 우선**

레이블은 원칙적으로 **명사형**으로 씁니다. 짧고 간결하게 필드의 내용만 지칭합니다.

| 🚫 문장형 (금지) | ✅ 명사형 (권장) |
|---------------|--------------|
| 이름을 입력해 주세요 | 이름 |
| 생년월일을 선택해 주세요 | 생년월일 |
| 이메일 주소를 입력하세요 | 이메일 주소 |

> **예외**: 복잡한 내용을 담은 필드이거나, 레이블 단독으로 의미 전달이 어려울 때는 문장형 사용 가능.
> 예: "현재 주소와 다른 주소로 배송받으시겠습니까?"

**원칙 2 — 필수·선택 표기 기준**

필수 항목에는 **(필수)** 표기를 레이블에 명시합니다. "선택" 표기는 선택 필드에만 최소한으로 사용합니다.

| 상황 | 표기 방식 | 예시 |
|------|---------|------|
| 필수 필드 | `레이블 (필수)` | 이름 (필수) |
| 선택 필드 | `레이블 (선택)` | 직장명 (선택) |
| 모든 필드 필수 시 | 상단 안내 후 `*` 사용 | "* 표시는 필수 항목입니다" |

> **금지**: `*` 기호만 단독 사용. 스크린 리더 사용자에게 전달되지 않습니다 — `aria-required="true"` 병행 필수.

---

#### 플레이스홀더 작성 원칙

**원칙 1 — 입력 예시 우선**

플레이스홀더는 "어떻게 입력해야 하는지"를 보여주는 **예시 형태**로 씁니다. 규칙·안내문은 레이블 아래 도움말 텍스트(hint text)로 분리합니다.

| 🚫 규칙 설명 (금지) | ✅ 입력 예시 (권장) |
|-----------------|----------------|
| 숫자만 입력하세요 | 예: 01012345678 |
| YYYY.MM.DD 형식으로 입력 | 예: 1990.01.15 |
| 영문 대소문자와 숫자 포함 | 예: Abc1234! |

**원칙 2 — 플레이스홀더로 레이블 대체 금지**

플레이스홀더는 입력 시 사라집니다. 레이블이 없는 필드는 접근성 위반입니다.

| 🚫 금지 패턴 | ✅ 올바른 패턴 |
|------------|-------------|
| 레이블 없이 `placeholder="이름"` 단독 사용 | 레이블: 이름 + `placeholder="예: 홍길동"` |
| 레이블 없이 `placeholder="이메일 입력"` | 레이블: 이메일 주소 + `placeholder="예: name@korea.go.kr"` |

**원칙 3 — 에러 상태 텍스트와 분리**

플레이스홀더가 사라진 자리에 에러 메시지를 표시하지 않습니다. 에러 텍스트는 필드 아래 별도 인라인 영역에 표시합니다. (4.1 에러 3단 구조 참조)

### 2.3 버튼 레이블 작성 기준

버튼은 주권자의 행동을 촉발하는 인터페이스의 핵심입니다. 레이블이 모호하면 클릭을 망설이고, 잘못된 레이블은 비가역적 실수로 이어집니다.

**핵심 원칙: 버튼은 '눌렀을 때 일어날 일'을 말합니다.**

---

#### 버튼 동사형 원칙

버튼 레이블은 반드시 **동사(행동)** 로 끝납니다. 명사형·형용사형 레이블은 행동을 표현하지 않습니다.

| 🚫 모호한 레이블 | ✅ 동사형 레이블 | 차이 |
|--------------|--------------|------|
| 확인 | 신청서 제출하기 | 무엇을 확인하는지 명확화 |
| 취소 | 작성 취소하고 나가기 | 취소의 결과 명시 |
| 다음 | 본인 인증으로 이동 | 다음 단계 예고 |
| 완료 | 신청 완료 — 접수번호 확인 | 결과 안내 포함 |
| 저장 | 변경 내용 저장하기 | 무엇을 저장하는지 명확화 |

> **글자 수 기준**: 버튼 레이블은 **12자 이내** 권장. 12자를 초과하면 핵심 동사를 앞에 두고 부연 설명은 생략합니다.

---

#### CTA 위계별 레이블 패턴

화면에 여러 버튼이 있을 때, 주요(Primary) CTA와 보조(Secondary) CTA의 레이블 위계를 구분합니다.

| 위계 | 목적 | 레이블 패턴 | 예시 |
|------|------|-----------|------|
| 주요 CTA (Primary) | 핵심 행동 완료 | [목적어] + [동사]하기 | 신청서 제출하기 |
| 보조 CTA (Secondary) | 되돌아가기 | [동사] + (장소/상태) | 이전 단계로 돌아가기 |
| 취소 (Tertiary) | 현재 작업 취소 | [작업] 취소하기 | 작성 취소하기 |
| 위험 CTA (Danger) | 비가역 삭제·초기화 | [목적어] [동사]하기 | 계정 삭제하기 |

---

#### 비가역 행동 앞 경고 문구 패턴

삭제·제출·결제 등 되돌릴 수 없는 행동이 따르는 버튼에는 반드시 경고 컨텍스트를 제공합니다.

**패턴 1 — 버튼 위 경고 텍스트**

```
[경고 텍스트] — 되돌릴 수 없는 이유를 한 문장으로
[위험 버튼]
```

예:
```
제출하면 내용을 수정할 수 없습니다. 작성 내용을 다시 확인해 주세요.
[신청서 최종 제출하기]
```

**패턴 2 — 확인 대화상자(Modal) 내 버튼 구성**

```
[대화상자 제목]: [목적어]을(를) [동사]하시겠습니까?
[본문]: 이 작업은 되돌릴 수 없습니다.
                 [취소 버튼]  [확인 동사 버튼]
```

예:
```
계정을 삭제하시겠습니까?
한 번 삭제된 계정과 데이터는 복원할 수 없습니다.
          [삭제 취소하기]  [계정 영구 삭제하기]
```

### 2.4 표기법·맞춤법 실무 기준 — 낡은 관공서체의 기계적 해체

공공 서비스의 텍스트는 친절을 빙자한 과도한 꾸밈이나, 책임을 회피하기 위한 모호한 표현을 철저히 배제해야 합니다. 주권자가 문장을 읽을 때 인지적 마찰을 겪지 않도록, 문장의 군더더기를 도려내는 기계적이고 단호한 표기 규칙을 적용합니다.

---

#### 🏛️ 2.4.1 과잉 존칭·피동형의 해체 — 권위를 뺀 조력자

정부 서비스의 고질병인 '책임 회피성 피동형'과 '과잉 의전'을 규칙으로 원천 차단합니다. 존중은 유지하되 텍스트는 압축합니다.

**선어말 어미 `-시-`의 전략적 제거**

공공기관은 친절해 보여야 한다는 강박 때문에 `-시-`를 남발하여 문장을 무겁게 만듭니다. 행위의 주체가 주권자 본인일 때는 `-시-`를 과감히 덜어내어 담백하게 안내합니다.

| 🚫 과잉 존칭 | ✅ 간결한 안내 |
|------------|------------|
| 증명서를 발급 받으신 후 가입하실 수 있습니다 | 증명서를 발급받은 후 가입할 수 있습니다 |
| 신청서를 제출하시기 바랍니다 | 신청서를 제출해 주세요 |
| 확인하시어 처리해 주시기 바랍니다 | 확인 후 처리해 주세요 |

**행정 주체의 명확화 — 피동형 금지**

`~되어지다`, `~처리되다` 같은 피동형은 행정기관이 책임을 피하려는 듯한 인상을 줍니다. 시스템이나 정부가 처리한 일은 능동형으로 명확히 선언합니다.

| 🚫 피동형 (책임 회피) | ✅ 능동형 (책임 있는 선언) |
|--------------------|----------------------|
| 신청이 완료되었습니다 | 신청을 완료했습니다 |
| 처리가 진행되고 있습니다 | 처리 중입니다 |
| 서류가 접수되었습니다 | 서류를 접수했습니다 |

> **예외**: 주권자에게 불이익이 가는 오류나 제한 상황에서만 충격을 완화하기 위해 피동형을 제한적으로 씁니다.
> `✅ (예외) 시스템 오류로 서비스 이용이 제한됩니다.`

---

#### 🏛️ 2.4.2 행정 부사·주관적 표현의 통제 — 객관적 길잡이

정부는 주권자의 상황을 평가하거나 감정을 섞어 안내하지 않습니다. 의미 없이 문장만 길어지게 만드는 주관적 부사를 엄격히 통제합니다.

**감정·속도 부사 금지**

`빠르게`, `이미`, `지금`, `너무` 등은 주권자마다 기준이 다른 주관적 단어입니다. 철저히 팩트(상태)만 전달합니다.

| 🚫 주관적 부사 | ✅ 객관적 사실 |
|-------------|------------|
| 신청 기한이 너무 많이 남지 않았습니다. 빠르게 제출해 주세요 | 신청 기한이 3일 남았습니다. 기한 내에 제출해 주세요 |
| 해당 민원은 이미 처리된 민원입니다 | 처리된 민원입니다 |
| 지금 바로 신청하세요 | 신청 기한: {날짜}까지 |

**금지 부사 목록**

| 금지 단어 | 금지 이유 | 대체 표현 |
|---------|---------|---------|
| 빠르게, 신속히 | 주관적 속도 기준 | 기한(날짜) 명시 |
| 이미, 벌써 | 중복 표현, 주관적 | 삭제 또는 상태값으로 대체 |
| 너무, 매우, 상당히 | 정도 기준 불명확 | 수치로 대체 |
| 간단히, 쉽게 | 주권자 경험에 따라 다름 | 단계 수로 대체 |
| 즉시, 곧 | 시간 불명확 | 예상 시간 명시 |

---

#### 🏛️ 2.4.3 숫자·행정 변수 표기 — 시각적 결속력 확보

숫자와 단위, 변수(이름, 날짜, 금액)가 텍스트 안에서 어떻게 결합하는지는 시각적 가독성과 직결됩니다.

**숫자와 단위의 밀착 표기**

주권자의 세금이나 기한과 직결되는 숫자와 단위는 반드시 붙여 써서 한눈에 덩어리(청크)로 읽히게 만듭니다.

| 🚫 띄어쓰기 (분리) | ✅ 밀착 표기 |
|----------------|-----------|
| 결제 금액: 15,000 원 | 결제 금액: 15,000원 |
| 기한: 30 일 | 기한: 30일 |
| 용량 제한: 10 MB | 용량 제한: 10MB |

**행정 변수(`{변수}`)의 조화**

이름, 부처명, 세금 등 시스템에서 불러오는 가변 데이터가 문장과 결합할 때는 조사의 어색함이 없도록 설계해야 합니다.

| 🚫 변수 위상 모호 | ✅ 시각적 대비 명확 |
|---------------|----------------|
| %s님이 %s원을 납부하셨습니다 | [홍길동] 님이 [15,000]원을 납부했습니다 |
| {이름}이 신청했습니다 | [{이름}]이(가) 신청했습니다 |

> **행정 변수 표기 규칙**: 시스템에서 불러오는 동적 데이터는 `[대괄호]`로 감싸 정적 텍스트와 시각적으로 구분합니다. 조사는 동적 값에 맞게 자동 처리하거나 `이(가)`, `을(를)` 형태로 병기합니다.

**보조 용언 띄어쓰기 통일**

공공기관 안내 문구에서 가장 많이 등장하는 보조 동사 `-주-`는 반드시 띄어 씁니다.

| 🚫 붙여 쓰기 | ✅ 띄어 쓰기 |
|-----------|-----------|
| 버튼을 눌러주세요 | 버튼을 눌러 주세요 |
| 확인해주세요 | 확인해 주세요 |
| 제출해주시기 바랍니다 | 제출해 주세요 |

---

#### 🏛️ 2.4.4 KRDS 맥락별 기호 위계 — 의미 있는 부호만 허용

장식용 특수문자를 남발하는 구시대적 게시판 문화를 버리고, 문장 부호에 명확한 '역할'을 부여합니다.

**UI 컴포넌트 vs 행정 상태값의 분리**

| 기호 | 역할 | 예시 |
|------|------|------|
| `[ ]` 대괄호 | 주권자가 직접 눌러야 하는 버튼·메뉴 경로 | [인증서 발급] 버튼을 눌러 주세요 |
| `' '` 작은따옴표 | 정부가 판정한 행정 상태·심사 결과 | 심사 상태가 '승인 대기'로 변경됩니다 |
| `( )` 소괄호 | 부가 설명·예시·선택 항목 | 숫자만 입력 (예: 01012345678) |
| `※` | 법적 필수 고지·중요 예외 사항 (최소화) | ※ 만 14세 미만은 법정대리인 동의 필요 |

**금지 기호 패턴**

| 🚫 금지 | 이유 | ✅ 대체 |
|--------|------|--------|
| ★, ◆, ● (장식 기호) | 시각적 노이즈, 스크린 리더 오독 | 제목 계층 구조 또는 볼드 사용 |
| → (화살표 남용) | 논리 관계 불분명 시 혼란 | 자연어로 연결 관계 명시 |
| !! 느낌표 연속 | 강조 목적이나 오히려 공격적 | 단일 느낌표도 오류 맥락에서 금지 |
| ~~취소선~~ (일반 안내) | 내용 삭제 의미로 혼동 | 텍스트 직접 제거 |


## 제3장. 정보핵심화의 원칙 — 실행 기준

정보핵심화는 '글자 수 줄이기'가 아닙니다. 주권자가 핵심에 도달하는 **생각의 거리**를 물리적으로 줄이는 작업입니다.

---

### 3.1 안내문·설명문 결론 우선 구조 (역피라미드)

공공 서비스의 안내문은 전통적으로 배경 → 과정 → 결론 순서로 전개됩니다. 그러나 주권자는 화면을 꼼꼼히 정독하지 않습니다. **결론을 찾기 위해 2문장 이상 읽어야 한다면 이미 실패입니다.**

---

#### 역피라미드 원칙: 핵심 먼저 → 상세 후에

```
┌─────────────────────────────────┐
│  핵심 결론 / 주권자가 취할 행동  │  ← 1문장, 맨 앞
├─────────────────────────────────┤
│       이유 / 조건 / 기준        │  ← 필요 시 2-3문장
├─────────────────────────────────┤
│  부연 설명 / 법적 근거 / 예외    │  ← 생략 가능, 접기 처리
└─────────────────────────────────┘
```

**Before (결론 후치):**
> 귀하의 소득 및 재산 현황, 가구원 수, 건강보험료 납부 실적 등을 종합 검토한 결과, 현재 귀하께서는 지원 요건 충족 기준에 미달하여 이번 지원 대상에 해당되지 않음을 알려드립니다.

**After (결론 우선):**
> **지원 대상이 아닙니다.**
> 소득 기준을 초과했습니다.
> 소득 인정액이 기준 중위소득 80% 이하여야 신청 가능합니다.

---

#### 40자 규칙: 긴 문장 줄이기

한 문장은 **40자 이내**를 기준으로 합니다. 40자를 초과하는 문장은 분리하거나 핵심만 남겨 재구성합니다.

| 초과 문장 | 분리 후 |
|---------|--------|
| 인터넷 연결이 불안정하거나 서버 응답이 지연되어 처리가 완료되지 않았을 수 있으니 잠시 후 다시 시도해 주시기 바랍니다. | 처리가 완료되지 않았습니다.<br>인터넷 연결을 확인하거나 잠시 후 다시 시도해 주세요. |
| 신청 완료 후 접수 결과 및 심사 진행 현황은 신청하신 이메일로 발송되는 알림 메시지를 통해 확인하실 수 있습니다. | 신청이 완료됐습니다.<br>심사 결과는 입력하신 이메일로 알려드립니다. |

---

#### 중요도 순서 설계 — 지금 vs. 나중

주권자가 지금 이 화면에서 필요한 정보와 나중에 참고할 정보를 구분하여 배치합니다.

| 구분 | 배치 위치 | 표현 방식 |
|------|---------|---------|
| 지금 필요한 정보 | 상단, 본문 첫 번째 | 굵게 또는 큰 폰트 강조 |
| 나중에 필요한 정보 | 하단, 접기(accordion) | 기본 숨김 처리 |
| 법적 필수 공지 | 하단, 작은 폰트 | 링크 또는 말줄임 처리 |

**설계 질문 3가지**

```
1. "이 화면에서 주권자가 가장 먼저 알아야 할 것은 무엇인가?"
   → 그것이 첫 문장이어야 한다.

2. "이 정보가 없어도 주권자가 다음 단계로 이동할 수 있는가?"
   → 없어도 된다면 하단 또는 접기 처리한다.

3. "40자를 넘는가?"
   → 넘는다면 핵심 1문장을 추출하고 나머지는 하위에 배치한다.
```

### 3.2 정보 구조 원칙 — 단어 수 제한과 청킹

단어를 줄이는 것만으로는 부족합니다. **어떻게 배치하느냐**가 주권자의 이해 속도를 결정합니다. 청킹(Chunking)은 관련된 정보를 의미 단위로 묶어 인지 부하를 줄이는 구조화 기술입니다.

---

#### 청킹의 3원칙

**원칙 1 — 한 문장 한 메시지**

한 문장에 하나의 의미만 담습니다. 의미의 마디(청크)가 3개를 넘으면 문장을 분리합니다.

| 🚫 과부하 문장 | ✅ 분리 후 |
|-------------|---------|
| 신청서 작성이 완료되면 제출 버튼을 눌러 주시고, 접수가 완료된 후에는 문자로 안내드릴 예정이오니 참고하여 주시기 바랍니다 | 신청서를 작성한 후 [제출하기]를 눌러 주세요.<br>접수가 완료되면 문자로 알려드립니다. |

**원칙 2 — 단계별 그룹핑**

긴 절차는 **단계(Step)** 단위로 명확히 분리합니다. 각 단계는 하나의 행동만 요청합니다.

```
✅ 신청 방법 (3단계)

Step 1. [본인 인증] 버튼을 눌러 주세요.
Step 2. 신청서를 작성하고 서류를 첨부해 주세요.
Step 3. [제출하기]를 눌러 신청을 완료해 주세요.
```

**원칙 3 — 조건과 결과의 분리**

조건문은 결과를 먼저, 조건을 다음에 배치합니다. (결론 우선 원칙 연계)

| 🚫 조건 우선 | ✅ 결론 우선 |
|-----------|-----------|
| 소득 기준 중위소득 80% 이하에 해당하는 경우 신청하실 수 있습니다 | 신청 가능합니다. 소득이 기준 중위소득 80% 이하여야 합니다. |
| 서류가 미비한 경우 신청이 반려될 수 있습니다 | 서류가 부족하면 신청이 반려됩니다. 아래 서류를 모두 준비해 주세요. |

|----------|-----------|------|

동일 위계의 항목은 품사와 구조를 통일합니다. 목록 안에서 명사형·동사형이 섞이면 안 됩니다.

### 3.3 알림·토스트·모바일 메시지 기준

화면 밖에서 도달하는 알림(푸시·SMS·이메일)과 화면 안의 즉시 피드백(토스트)은 정보 핵심화 원칙의 가장 극단적인 적용 영역입니다. 공간과 주의 시간이 가장 짧기 때문에, 불필요한 말은 단 한 글자도 허용하지 않습니다.

---

#### 모달 팝업 (치명적 오류·경고)

느낌표(!)를 쓰지 않고, 건조한 톤으로 **[원인 + 해결책]**을 가장 짧은 청크로 끊어 던집니다. 주어(시스템)의 책임을 주권자에게 전가하지 않습니다.

| 🚫 잘못된 모달 | ✅ 올바른 모달 |
|-------------|------------|
| 입력하신 정보가 잘못되었습니다! 다시 확인해주세요. | 인증 번호가 틀렸습니다.<br>문자로 받은 6자리 숫자를 다시 입력해 주세요. |
| 오류가 발생했습니다. 죄송합니다. | 처리할 수 없습니다.<br>잠시 후 다시 시도해 주세요. |

**모달 구조 원칙**

```
제목: [상황 요약 — 한 줄, 명사형]
본문: [원인 or 상세 설명 — 최대 2문장]
버튼: [주 행동] + [보조 행동 — 선택]
```

---

#### 토스트 알림 (가벼운 상태 피드백)

`복사 완료`, `저장 완료` 등 주권자의 행동이 정상적으로 접수되었음을 **1-2초 내**에 알립니다. 서술어를 생략한 **명사형**으로 짧게 칩니다.

| 🚫 너무 긴 토스트 | ✅ 간결한 토스트 |
|--------------|-------------|
| 클립보드에 계좌번호가 성공적으로 복사되었습니다 | 계좌번호를 복사했습니다 |
| 변경 사항이 저장되었습니다 | 저장 완료 |
| 신청이 완료되어 접수번호가 발급되었습니다 | 신청 완료. 접수번호: 2026-04-18-0042 |

**토스트 글자 수 기준**

| 상황 | 최대 글자 수 | 예시 |
|------|-----------|------|
| 단순 성공 | 10자 이내 | 저장 완료 |
| 접수 확인 | 20자 이내 | 신청 완료. 접수번호: {N} |
| 오류 피드백 | 30자 이내 | 업로드 실패. 파일 크기 초과 |

---

#### 푸시 알림·SMS 기준

모바일 알림은 잠금 화면에서 맥락 없이 노출됩니다. **서비스명 + 핵심 정보 + 행동 버튼** 구조를 따릅니다.

**푸시 알림 구조**

```
[서비스명] 핵심 내용 (25자 이내)
상세 설명 (선택, 50자 이내)
[행동 버튼]
```

| 🚫 맥락 없는 알림 | ✅ 명확한 알림 |
|--------------|-------------|
| 새로운 알림이 있습니다 | [복지로] 기초생활 지원금 신청 결과가 나왔습니다 |
| 마감이 다가오고 있습니다 | [홈택스] 종합소득세 신고 마감 3일 전 — 5월 31일까지 |

**SMS 기준**

| 항목 | 기준 |
|------|------|
| 발신자 표시 | `[기관명]` 또는 공식 번호 — 스팸 의심 방지 |
| 첫 문장 | 수신자에게 관련된 핵심 정보 (결론 우선) |
| 링크 | 단축 URL 사용 시 도메인 명시 (`go.kr`, `gov.kr`) |
| 길이 | 90바이트 이내 권장 (1건 처리) |


## 제4장. 심리적안전망의 원칙 — 실행 기준

공공 서비스에서 오류는 반드시 발생합니다. 문제는 오류 자체가 아니라, 오류를 만났을 때 시민이 어떤 경험을 하느냐입니다.

심리적안전망 원칙의 실행 기준은 사용자가 실수하거나 어려움에 처했을 때 언제든 궤도를 수정할 수 있다는 확신을 주는 텍스트 패턴을 정의합니다.

---

### 4.1 에러 메시지 3단 구조

#### 왜 3단인가

기존 공공 서비스 오류 메시지의 두 가지 실패 유형:

**유형 A — 코드만 던지는 메시지**
> `ERROR 4023: 인증 실패`

원인도 해결책도 없다. 시민은 무엇을 해야 할지 모른 채 화면 앞에 멈춘다.

**유형 B — 사용자를 탓하는 메시지**
> `입력하신 정보가 올바르지 않습니다. 다시 확인해 주세요.`

무엇이 올바르지 않은지 알려주지 않으면서 시민의 잘못으로 전제한다. 당혹감과 분노를 동시에 준다.

**3단 구조는 이 두 실패를 모두 막는다.** 무슨 일이 있었는지 알려주고, 왜 그런지 설명하고, 무엇을 하면 되는지 제시한다.

---

#### 3단 구조 정의

```
1단 — 상황 (What): 무슨 일이 있었는지
2단 — 원인 (Why): 왜 그런지 (시스템 측 원인 우선, 사용자 탓 금지)
3단 — 행동 (How): 지금 무엇을 하면 되는지
```

| 단계 | 역할 | 작성 원칙 |
|------|------|----------|
| 1단. 상황 | 무슨 일이 일어났는지 한 문장으로 | 시스템 관점이 아닌 사용자 관점으로 서술 |
| 2단. 원인 | 왜 그런 일이 생겼는지 | 시스템·환경 원인 우선. 사용자 입력 문제라도 "확인 필요"로 표현 |
| 3단. 행동 | 지금 할 수 있는 구체적 행동 | 버튼·링크 포함. 2개 이상 대안 제시 가능 |

> **3단 모두 필요한가?** 맥락에 따라 1+3단(원인 불필요한 경우), 1+2단(즉각 행동 없는 경우)으로 축약 가능. 단, 1단(상황)은 항상 필수.

---

#### 3단 구조 적용 예시

**예시 1 — 인증 번호 불일치**

```
❌ ERROR 4023: 인증 실패
❌ 인증에 실패했습니다. 다시 시도해 주세요.

✅ 1단 인증 번호가 일치하지 않습니다.
   2단 문자로 받은 6자리 숫자와 다르게 입력됐습니다.
   3단 문자를 다시 확인하고 숫자를 입력해 주세요.
      번호가 만료됐다면 [인증 번호 다시 받기]를 눌러 주세요.
```

**예시 2 — 파일 업로드 실패**

```
❌ 파일 업로드에 실패했습니다.
❌ 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.

✅ 1단 파일을 올리지 못했습니다.
   2단 파일 크기가 10MB를 넘으면 올릴 수 없습니다.
   3단 파일 크기를 줄이거나, 여러 개로 나눠 올려 주세요.
      (지원 형식: PDF, JPG, PNG / 최대 10MB)
```

**예시 3 — 세션 만료**

```
❌ 세션이 만료되었습니다. 로그인 페이지로 이동합니다.

✅ 1단 자리를 비운 사이 로그인이 끊겼습니다.
   2단 보안을 위해 30분 이상 사용하지 않으면 자동으로 로그아웃됩니다.
   3단 작성 중이던 내용은 임시 저장됐습니다. 다시 로그인하면 이어서 작성하실 수 있습니다.
      [다시 로그인하기]
```

**예시 4 — 검색 결과 없음**

```
❌ 검색 결과가 없습니다. 철자나 맞춤법 오류가 있는지 확인해 주세요.

✅ 1단 '{검색어}'에 대한 결과를 찾지 못했습니다.
   (2단 생략 — 원인이 다양하고 특정이 어려움)
   3단 단어를 짧게 줄여 보세요. (예: '행정 절차 안내' → '절차')
      비슷한 표현으로 다시 검색해 보세요.
```

**예시 5 — 필수 항목 미입력 (폼 유효성 검사)**

```
❌ 필수 항목을 입력해 주세요.
❌ 잘못된 입력입니다.

✅ 1단 '생년월일'을 입력하지 않았습니다.
   (2단 생략 — 원인이 명확한 경우)
   3단 생년월일을 숫자 8자리로 입력해 주세요. (예: 19900101)
```

---

#### 금지 표현 목록

아래 표현은 사용자를 탓하거나, 정보 없이 재시도를 요구하거나, 심리적 부담을 주는 패턴입니다.

| 금지 표현 | 금지 이유 | 대체 방향 |
|----------|----------|----------|
| 잘못 입력하셨습니다 | 사용자 실수로 단정 | "일치하지 않습니다" / "확인이 필요합니다" |
| 올바르지 않은 정보입니다 | 무엇이 올바르지 않은지 미제시 | 구체적 필드명 + 기대 형식 명시 |
| 오류가 발생했습니다 (단독) | 원인·해결책 없는 빈 메시지 | 3단 구조 적용 |
| 다시 시도해 주세요 (단독) | 무엇을 어떻게 재시도해야 하는지 불명 | 구체적 행동 제시 |
| 맞춤법 오류가 있는지 확인해 주세요 | 사용자 탓 전제 | "결과를 찾지 못했습니다" + 검색 팁 |
| 철자를 확인해 주세요 | 사용자 탓 전제 | 동일 |
| 시스템 오류입니다 (단독) | 다음 행동 없음 | 담당 부서 연락처 또는 재시도 안내 |
| ERROR [코드] | 시민에게 의미 없는 기술 정보 | 1단 상황 설명으로 대체 (코드는 괄호 안에 선택 병기) |
| 잠시 후 다시 시도해 주세요 (단독) | '잠시'가 얼마인지 불명 | "5분 후 다시 시도해 주세요" 또는 담당자 연락처 안내 |
| 죄송합니다 (오류 메시지에서) | 공감처럼 보이나 원인·해결책을 가림 | 공감 없이 건조하게 상황+행동만 제시 |

> **주의**: 오류 메시지에서 이모지나 느낌표(!) 사용 금지. 오류 상황에서 과한 감정 표현은 조롱처럼 느껴질 수 있다. (1.4 보이스·톤 기준 — 🔴 톤 3 참조)

---

#### 원인 귀속 원칙 — 사용자를 탓하지 않는 언어

오류의 원인은 세 가지입니다: 시스템, 환경, 사용자 입력. 실제 원인이 사용자 입력이더라도, 언어는 다음 원칙을 따릅니다.

**원칙 1 — 시스템·환경 원인은 명확히 밝힌다**

> ✅ 서버가 응답하지 않고 있습니다. (시스템 원인)
> ✅ 인터넷 연결을 확인해 주세요. (환경 원인)

**원칙 2 — 사용자 입력 문제는 "확인 필요" 방식으로**

"틀렸다"가 아니라 "일치하지 않는다", "형식을 확인해 달라"로 표현한다.

| 사용자 탓 표현 | 중립 표현 |
|-------------|---------|
| 비밀번호가 틀렸습니다 | 비밀번호가 일치하지 않습니다 |
| 잘못 입력한 항목이 있습니다 | 확인이 필요한 항목이 있습니다 |
| 형식에 맞지 않는 값입니다 | 입력 형식을 확인해 주세요 (예: 숫자 8자리) |

**원칙 3 — 원인을 모를 때는 솔직하게**

원인 불명 시 억지로 이유를 만들지 않는다. "일시적인 문제"로 표현하고 연락처를 제공한다.

> ✅ 일시적인 문제가 생겼습니다. 잠시 후 다시 시도하거나, 문제가 계속되면 고객센터(☎ 1234-5678)에 문의해 주세요.

---

#### 오류 유형별 빠른 참조

| 오류 유형 | 1단 (상황) | 2단 (원인) | 3단 (행동) |
|----------|-----------|-----------|-----------|
| 로그인 실패 | 아이디 또는 비밀번호가 일치하지 않습니다 | — | 다시 입력하거나 [비밀번호 찾기] |
| 세션 만료 | 로그인이 끊겼습니다 | 30분 이상 사용하지 않으면 자동 로그아웃 | [다시 로그인하기] |
| 파일 크기 초과 | 파일을 올리지 못했습니다 | {N}MB를 넘는 파일은 올릴 수 없습니다 | 파일 크기를 줄이거나 나눠서 올려 주세요 |
| 필수 항목 미입력 | '{항목명}'을 입력하지 않았습니다 | — | {형식} 형태로 입력해 주세요 |
| 네트워크 오류 | 연결이 끊겼습니다 | 인터넷 연결 상태가 불안정합니다 | 연결 확인 후 새로 고침 해 주세요 |
| 검색 결과 없음 | '{검색어}'에 대한 결과를 찾지 못했습니다 | — | 검색어를 바꿔 다시 검색해 보세요 |
| 서버 오류 | 일시적인 문제가 생겼습니다 | — | 잠시 후 다시 시도하거나 고객센터 문의 |
| 권한 없음 | 이 페이지에 접근할 수 없습니다 | 로그인이 필요하거나 접근 권한이 없습니다 | [로그인하기] 또는 담당자 문의 |
| 중복 입력 | 이미 신청하셨습니다 | 같은 내용으로 신청한 이력이 있습니다 | [신청 내역 보기] |
| 시간 초과 | 시간이 초과되었습니다 | 제출 가능 시간({N}분)이 지났습니다 | 처음부터 다시 시작해 주세요 |

### 4.2 성공·완료 메시지 패턴

과업이 완료됐을 때의 메시지는 **주권자의 안도감을 강화**하고 **다음 일정을 명확히** 안내하는 역할을 합니다. 과도한 축하나 감사 표현은 오히려 공식 서비스의 신뢰감을 해칩니다.

---

#### 완료 메시지 3요소

```
① 완료 확인 — 무엇이 완료됐는지 (1문장)
② 접수 정보 — 접수번호·다음 처리 일정
③ 다음 행동 안내 — 버튼 또는 링크 (선택)
```

| 요소 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| ① 완료 확인 | 감사합니다! 신청해 주셔서 정말 기쁩니다. | 지원금 신청이 완료됐습니다. |
| ② 접수 정보 | 빠른 시일 내에 연락드리겠습니다. | 심사 결과는 3영업일 안에 문자로 알려드립니다. |
| ③ 다음 행동 | — | [신청 내역 확인하기] |

---

#### 완료 메시지 유형별 패턴

**유형 1 — 즉시 처리 완료 (실시간 시스템 처리)**

```
✅ {작업}이 완료됐습니다.
   접수번호: {N}
   [결과 확인하기]
```

예시:
```
✅ 비밀번호가 변경됐습니다.
   다음 로그인부터 새 비밀번호로 접속해 주세요.
   [로그인하기]
```

**유형 2 — 심사 대기형 (행정 처리 필요)**

```
✅ {신청명} 신청이 완료됐습니다.
   접수번호: {N}
   심사 결과는 {기간} 안에 {수신 방법}으로 알려드립니다.
   [신청 내역 확인하기]
```

예시:
```
✅ 기초생활 지원금 신청이 완료됐습니다.
   접수번호: 2026-복지-004821
   심사 결과는 14일 안에 등록하신 휴대전화로 알려드립니다.
   [신청 내역 확인하기]
```

**유형 3 — 취소·해지 완료 (되돌릴 수 없는 행동)**

```
✅ {서비스명} {취소/해지}이 완료됐습니다.
   {취소 정보 또는 효력 발생 일자}
   [확인서 내려받기]   (선택)
```

예시:
```
✅ 자동 납부 해지가 완료됐습니다.
   다음 달 청구분부터 자동 납부가 되지 않습니다.
   수동으로 납부하려면 납부 기한(매월 25일)을 확인해 주세요.
```

---

#### 완료 메시지 금지 표현

| 🚫 금지 표현 | 금지 이유 |
|-----------|---------|
| 감사합니다! 다음에도 이용해 주세요 | 과업 완료 확인보다 마케팅 메시지가 앞섬 |
| 축하드립니다🎉 | 공공 서비스에서 과도한 감정 표현 |
| 처리 완료! 수고하셨습니다 | 주권자를 칭찬하는 것은 권위적 태도 |
| 곧 연락드리겠습니다 | '곧'의 시간 불명확 → 구체적 기간 명시 |
| 오류 없이 완료됐습니다 | "오류 없이"는 당연한 것 → 삭제 |

### 4.3 빈 상태·검색 없음 텍스트

데이터가 없는 빈 화면은 오류가 아닙니다. 그러나 "내역이 없습니다"라는 건조한 통보만 덩그러니 남겨두면 주권자는 자신이 무언가 잘못한 것은 아닌지 불안해하거나 다음 행동을 멈추게 됩니다.

---

#### 빈 상태(Empty State) 구조 원칙

```
① 상태 설명 — 왜 비어 있는지 (원인)
② 안내 — 무엇을 하면 되는지 (대안 행동)
③ 행동 버튼 — 직접 연결되는 CTA (선택)
```

| 상황 | 🚫 나쁜 예 | ✅ 좋은 예 |
|------|---------|---------|
| 신청 내역 없음 | 신청 내역이 없습니다. | 아직 신청한 지원금이 없습니다.<br>[나에게 맞는 지원금 찾기] |
| 알림 없음 | 알림이 없습니다. | 새로운 알림이 없습니다.<br>신청·심사·결과 변경 시 알려드립니다. |
| 즐겨찾기 없음 | 즐겨찾기가 없습니다. | 자주 쓰는 서비스를 즐겨찾기에 추가해 보세요.<br>[서비스 둘러보기] |

---

#### 빈 상태 유형별 패턴

**유형 1 — 아직 생성되지 않은 경우**

주권자가 행동하면 채워질 수 있는 상태입니다. 대안 행동을 유도합니다.

```
아직 {내용}이 없습니다.
[{생성/신청/추가} 버튼]
```

**유형 2 — 조건을 충족하지 못해 비어 있는 경우**

주권자가 해당되지 않아 표시될 내용이 없는 상태입니다. 이유를 설명합니다.

```
현재 {조건}에 해당하는 {내용}이 없습니다.
{조건 변경 또는 다른 경로 안내}
```

예시:
```
현재 신청 가능한 지원금이 없습니다.
소득·재산 기준이 맞지 않거나, 이미 신청한 경우입니다.
[지원금 자격 다시 확인하기]
```

**유형 3 — 오류로 인해 데이터를 불러오지 못한 경우**

빈 상태처럼 보이지만 실제로는 기술적 문제입니다. 오류 3단 구조를 적용합니다.

```
정보를 불러오지 못했습니다.
잠시 후 [새로 고침]하거나, 문제가 계속되면 고객센터에 문의해 주세요.
[새로 고침] [고객센터 문의]
```

---

#### 검색 없음 (No Results) 패턴

단순히 "결과가 없다"고 끝내지 않고, 주권자가 검색을 성공할 수 있도록 힌트를 제공합니다.

**기본 패턴**

```
'{검색어}'에 대한 결과를 찾지 못했습니다.
— 단어의 철자가 맞는지 확인해 보세요.
— 단어를 짧게 줄여 보세요. (예: '행정 절차 안내' → '절차')
— 비슷한 표현으로 다시 검색해 보세요.
```

**유형별 힌트**

| 검색 유형 | 힌트 방향 | 예시 |
|---------|---------|------|
| 서비스 검색 | 대표 카테고리 제안 | "'청년주택'을 찾고 계신가요? [청년 지원 서비스 보기]" |
| 기관명 검색 | 공식 명칭 안내 | "'서울시청'으로 검색해 보세요" |
| 민원 검색 | 인기 검색어 제안 | "많이 찾는 민원: 주민등록등본, 건강보험료 조회" |

---

#### 검색창 플레이스홀더 — 실패 방지 힌트

검색창의 입력 안내 문구(플레이스홀더)는 "무엇을 입력하면 원하는 결과를 얻을 수 있는지" 보여주는 가장 좋은 튜토리얼입니다.

| 🚫 의미 없는 안내 | ✅ 구체적 힌트 |
|--------------|------------|
| 검색어를 입력해 주세요 | 질병명 또는 병원 이름 검색 |
| 검색 | 예) 청년 주택, 자동차세 연납 |
| 찾고 싶은 내용을 입력하세요 | 서비스명, 기관명, 민원 유형 |


## 제5장. 기관별 파생 가이드 템플릿 (Layer 3)

> 이 장은 KRDS UX Writing 가이드(Layer 2)를 개별 기관의 서비스·조직 특성에 맞게 자가 적용할 때 사용합니다.
> **목표**: 기관마다 가이드를 처음부터 만들지 않고, 검증된 구조 위에서 **맞춤화**합니다.

---

### 5.1 파생 가이드란 무엇인가

#### 3계층 구조: 선언 → 실행 → 적용

공공기관 UX 라이팅은 세 가지 레이어가 쌓이는 구조로 운영됩니다.

```
┌─────────────────────────────────────────────┐
│  Layer 1 — KRDS (선언)                       │
│  "명확하게·간결하게·유용하게·일관되게"       │
│  → 원칙의 방향만 선언, 구체 기준 없음        │
├─────────────────────────────────────────────┤
│  Layer 2 — 이 가이드 (실행)                  │
│  3대 원칙·용어 사전·에러 구조·체크리스트     │
│  → KRDS 선언을 실무 행동 기준으로 번역       │
├─────────────────────────────────────────────┤
│  Layer 3 — 기관별 파생 가이드 (적용)         │
│  기관 전용 행정어·톤 정의·에러 시나리오      │
│  → Layer 2를 해당 기관 맥락에 맞게 맞춤화   │
└─────────────────────────────────────────────┘
```

**핵심 원칙**: Layer 3는 Layer 2를 **대체하지 않습니다**. 공통 규칙은 그대로 상속하고, 기관 고유의 공백만 채웁니다.

| 구분 | Layer 2 (공통) | Layer 3 (파생) |
|------|---------------|---------------|
| 3대 원칙 | 상속 (변경 불가) | 상속 (변경 불가) |
| 행정어 사전 | 150개 공통 | 기관 전문어 10~30개 추가 |
| 톤·매너 | 4단계 매트릭스 제공 | 기관별 주력 톤 선택·정의 |
| 에러 메시지 | 3단 구조 제공 | 기관 주요 오류 5개 작성 |
| 체크리스트 | 23개 항목 제공 | 기관 적용 항목 선택·조정 |

#### SeMA 파생 사례 — 문화예술 기관 적용 예시

서울시립미술관(SeMA)은 Layer 2 가이드를 기반으로 문화예술 기관 특성을 반영한 Layer 3 파생 가이드를 적용했습니다.

**① 전문용어 추가 사전 (일부)**

| 원어 | 대체어 | 맥락 | 비고 |
|------|--------|------|------|
| 아카이브 | 자료실 | 작품·문서 보관 안내 | '소장품 검색'으로도 병행 사용 |
| 큐레이션 | 기획·선별 | 전시 콘텐츠 설명 | 전문가 대상 문서에선 원어 허용 |
| 오프닝 리셉션 | 개막 행사 | 행사 예약 화면 | 초대권 안내에 한해 원어 허용 |
| 도슨트 | 전시 해설사 | 예약·결제 화면 | 첫 등장 시 괄호 병기: 도슨트(전시 해설사) |

**② 기관 특화 톤 선택**

SeMA는 4단계 매트릭스(공식/표준/친근/긴급) 중 **표준·친근** 두 단계를 주력으로 채택했습니다.

| 상황 | 적용 톤 | 근거 |
|------|---------|------|
| 전시 안내·설명 | 친근 | 관람 경험 전 진입장벽 낮추기 |
| 예약·결제·취소 | 표준 | 거래 신뢰성 확보 |
| 긴급 공지 (휴관 등) | 긴급 | 즉시 인지 필요 |
| 소장품·아카이브 | 표준 | 정보 전달 우선 |

**③ 에러 메시지 기관 적용 사례**

> 예약이 완료되지 않았습니다.
> 선택하신 시간대가 마감되었습니다.
> 다른 시간대를 선택하거나 [대기 신청하기]를 눌러 주세요.

→ 3단 구조(무슨 일 / 왜 / 어떻게) + SeMA 특화 CTA("대기 신청하기") 적용

---

### 5.2 파생 가이드 제작 체크리스트

새로운 기관이 Layer 3 파생 가이드를 만들 때 순서대로 완료합니다.

```
□ Step 1. 기관 핵심 서비스 파악
   ㄴ 주요 사용자 흐름 3가지 이상 열거 (민원신청/예약/콘텐츠 조회 등)
   ㄴ 흐름별 핵심 화면 목록 작성

□ Step 2. 기관 고유 행정어·전문용어 추가 목록 작성
   ㄴ 최소 10개, 권장 20~30개
   ㄴ 양식: 5.3 ① 기관별 행정어 추가 사전 사용

□ Step 3. 기관별 톤·매너 정의
   ㄴ 4단계 매트릭스(공식/표준/친근/긴급) 중 주력 2개 선택
   ㄴ 상황별 적용 기준 표 완성 (최소 4가지 상황)
   ㄴ 양식: 5.3 ② 기관별 톤 결정 가이드 사용

□ Step 4. 주요 에러 시나리오 5개 선별 + 3단 구조 적용
   ㄴ 해당 서비스에서 가장 자주 발생하는 오류 5개 선택
   ㄴ 각각에 "무슨 일 / 왜 / 어떻게" 3단 구조 작성
   ㄴ 4.1 에러 메시지 3단 구조 참조

□ Step 5. 자가진단 체크리스트 기관 맞춤 수정
   ㄴ 부록 B 23개 항목 중 기관 서비스에 해당하는 항목 선택
   ㄴ 기관 전용 항목 최대 5개 추가
```

**완료 기준**: 5개 Step 모두 체크 시 Layer 3 파생 가이드 초안 완성으로 간주합니다.

---

### 5.3 파생 가이드 빈 양식 — 바로 복사해서 쓰는 템플릿

아래 세 가지 양식을 복사해 기관별 파생 가이드 문서에 붙여넣고 작성합니다.

#### ① 기관별 행정어·전문용어 추가 사전

**이 사전의 금지 원칙**
- 대체어도 생소한 경우: 괄호 병기 후 3회 이상 등장 시 단독 사용
- 법령 조문에 명시된 용어: 초회 등장 시 괄호 병기, 이후 일상어 대체 허용

#### ② 기관별 톤 결정 가이드

> **작성 가이드**

> - '상황' 열에 기관 서비스 맥락을 구체적으로 기재합니다.

**기관명**: ________________________________

**주력 톤**: □ 공식  □ 표준  □ 친근  □ 긴급  (2개 선택)

| 상황 | 적용 톤 | 샘플 문장 | 금지 표현 |
|------|---------|---------|---------|
| (예: 민원 접수 확인) | 표준 | "접수가 완료되었습니다." | "처리되셨습니다" |

1. 법적 효력이 있는 안내인가? → 공식
2. 일상적 거래·조회·예약인가? → 표준

> **작성 가이드**
> - '주제' 열: 어떤 UX 상황·컴포넌트인지 (예: 버튼 레이블, 에러 메시지)

> - 'Do' 열: 3대 원칙을 적용한 개선 표현
> - '적용 원칙' 열: 무번역/정보핵심화/심리적안전망 중 해당 원칙 기재

| 주제 | Don't | Do | 적용 원칙 |
|------|-------|----|---------|
| (예: 제출 버튼) | 제출하기 | 신청서 보내기 | 무번역 |

## 제6장. UI 컴포넌트 라이팅 — 화면을 조립하는 원자 단위 글쓰기

> **이 장의 포지셔닝**: 아무리 훌륭한 정책이라도 버튼의 단어 하나가 모호하면 주권자는 길을 잃습니다. KRDS 공식 UI 컴포넌트 14종에 적용해야 하는 가장 기계적이고 명확한 글쓰기 공식을 정리합니다.

---

### 6.1 버튼 (Buttons): 다음 화면을 100% 예측하게 하라

버튼은 주권자가 과업을 완수하기 위해 통과해야 하는 '문'입니다. 문 너머에 무엇이 있는지 예측할 수 없다면 불안감을 줍니다.

**규칙 1 — 관성적인 '확인/취소' 금지**

문맥 없이 쓰이는 '확인'이나 '다음'은 어떤 결과가 일어날지 알려주지 않습니다. 반드시 **[목적어 + 동사형 명사]** 구조로 명확한 행동을 지시합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 확인 | 지원금 신청하기 |
| 다음 단계로 | 입력 정보 확인하기 |

**규칙 2 — 부정적 행동의 명확화**

'취소', '삭제', '포기' 등 파괴적 결과를 낳는 버튼(Secondary/Text Button)일수록 단호하게 씁니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 돌아가기 | 신청 취소하기 |
| 다음에 하기 | 작성 내용 삭제하기 |

---

### 6.2 입력 폼 (Text Input & Helper Text): 인지적 마찰을 없애라

입력창은 주권자가 정부에 자신의 데이터를 건네주는 가장 예민한 영역입니다. **3중 구조**로 안내합니다.

**① 레이블 (Label)**
폼의 이름은 가장 보편적인 명사로 짧게 씁니다.
- ✅ `휴대전화 번호`

**② 플레이스홀더 (Placeholder)**
입력창 안의 회색 텍스트는 '질문'이 아니라 **'입력 예시'나 '형식'**을 보여주어야 합니다. 플레이스홀더를 레이블 대신 사용하면 절대 안 됩니다(입력 시 사라지므로 맥락 상실).

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 전화번호를 입력해 주세요. | 숫자만 11자리 입력 |

**③ 안내 문구 (Helper Text)**
오류가 나기 전에 제약 조건(글자 수, 특수문자 여부 등)을 입력창 바로 아래에 **선제적으로** 제공합니다.
- ✅ 입력창 하단: `하이픈(-)을 제외하고 입력해 주세요.`

---

### 6.3 선택지 (Checkbox & Radio Button): 긍정문으로 묻고, 상호 배타적으로 구성하라

체크박스(다중 선택)와 라디오 버튼(단일 선택)은 주권자의 의사를 확정하는 도구입니다. '다크 패턴(Dark Pattern)'을 철저히 금지합니다.

**규칙 1 — 이중 부정 금지**

"수신을 거부하시겠습니까?"에 "아니오"를 누르게 하는 방식은 금지합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 야간 알림 수신을 거부하지 않습니다. | 야간 알림 수신 동의 |

**규칙 2 — 항목의 상호 배타성**

라디오 버튼의 항목들은 서로 의미가 겹치지 않도록 명확하게 잘라내야 합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 20대 / 20~30대 / 30대 이상 | 20대 이하 / 30대 / 40대 이상 |

---

### 6.4 모달 팝업과 토스트 (Modal & Toast): 감정을 빼고 해결책을 던져라

**모달 팝업 (치명적 오류 및 경고)**

느낌표(!)를 쓰지 않고, 건조한 톤으로 **[원인 + 해결책]**을 가장 짧은 청크로 끊어 던집니다. 주어(시스템)의 책임을 주권자에게 전가하지 않습니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 입력하신 정보가 잘못되었습니다! 다시 확인해주세요. | 인증 번호가 틀렸습니다. 문자로 받은 6자리 숫자를 다시 입력해 주세요. |

**토스트 알림 (가벼운 상태 피드백)**

주권자의 행동이 정상적으로 접수되었음을 1~2초 내에 알립니다. 서술어를 생략한 명사형으로 짧게 씁니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 클립보드에 계좌번호가 성공적으로 복사되었습니다. | 계좌번호를 복사했습니다. / 복사 완료 |

---

### 6.5 아코디언과 툴팁 (Accordion & Tooltip): 맥락을 지키는 정보의 숨바꼭질

**툴팁 (ⓘ)**

행정 전문 용어를 번역할 때 사용합니다. 툴팁 안의 텍스트는 **두 문장을 넘지 않도록** 간결하게 작성합니다.

> 예: `과세표준액 ⓘ` → (터치 시) `세금을 매기는 기준이 되는 금액입니다.`

**아코디언 (접기/펴기)**

예외 조항이나 상세 법적 근거를 숨길 때 씁니다. 접혀 있을 때 노출되는 헤더(Title)는 이 안에 어떤 정보가 있는지 명확히 유추할 수 있어야 합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 더 알아보기 ∨ | 해외 체류자 대리 신청 방법 ∨ |

---

### 6.6 빈 화면 (Empty State): 막다른 길에 표지판 세우기

데이터가 없는 빈 화면은 오류가 아닙니다. 하지만 "내역이 없습니다"라는 건조한 통보만 남겨두면 주권자는 불안해하거나 행동을 멈추게 됩니다.

**규칙 1 — 원인과 대안의 결합**

왜 비어 있는지 **이유(원인)**를 설명하고, 다음에 무엇을 해야 하는지 **행동(대안)**을 버튼이나 링크로 반드시 함께 제시합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 신청 내역이 없습니다. (막다른 길) | 아직 신청한 지원금이 없습니다. [나에게 맞는 지원금 찾기] 버튼 제공 |

**규칙 2 — 검색 결과 없음**

단순히 "결과가 없다"고 끝내지 않고, 주권자가 검색을 성공할 수 있도록 팁을 제공합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 검색 결과가 없습니다. | '지원금'에 대한 검색 결과가 없습니다. 단어의 철자가 맞는지 확인하거나, 비슷한 다른 단어로 다시 검색해 보세요. |

---

### 6.7 단계 표시기 (Stepper/Progress): 현재 위치와 남은 거리의 명확화

세금 신고, 지원금 신청처럼 호흡이 긴 공공 서비스 과업에서는 '내가 지금 어디에 있고, 앞으로 얼마나 남았는지'를 투명하게 알려주어야 이탈률을 낮출 수 있습니다.

**규칙 1 — 숫자와 명사의 결합 (프런트 로딩)**

긴 서술어를 버리고, **[숫자/전체 + 행동 명사]** 형태로 압축합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 두 번째 단계: 기본 정보를 입력해 주세요 | Step 2/4. 기본 정보 입력 |

**규칙 2 — 완료된 단계와 예정된 단계의 위계 분리**

이미 지나온 단계는 완료(과거형/명사형)로, 현재는 진행 중임을 시각적 텍스트로 구분합니다.

> 예시: `✔ 약관 동의 → 2. 정보 입력 (현재) → 3. 서류 첨부`

---

### 6.8 태그와 배지 (Tag & Badge): 상태와 속성의 직관적 요약

목록에서 특정 항목의 상태(진행 중, 완료 등)나 속성(필수, 선택, 새로운 기능)을 눈에 띄게 알려주는 작은 이름표입니다.

**규칙 1 — 최대 4음절의 명사형 압축**

배지 안의 텍스트는 2~4글자의 명사형으로 잘라냅니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 심사가 진행 중입니다 | 심사 중 |
| 새로 올라온 공지 | 새 글 / NEW |

**규칙 2 — 공공 서비스 상태 위계 표준**

| 배지 | 의미 |
|------|------|
| `접수 완료` | 신청서가 정상적으로 제출됨 |
| `심사 중` | 담당자가 서류를 검토하고 있음 |
| `보완 요청` | 주권자가 서류를 다시 내야 함 (행동 필요) |
| `승인 완료` | 최종 결과 확정 (긍정) |
| `반려` | 최종 결과 확정 (부정) |

---

### 6.9 검색창 (Search Bar): 실패를 방지하는 힌트

검색창의 플레이스홀더는 "여기에 무엇을 입력해야 원하는 결과를 얻을 수 있는지"를 보여주는 가장 좋은 튜토리얼입니다.

**규칙 — 대표 검색어 예시 제공**

"검색어를 입력하세요" 같은 하나 마나 한 소리 대신, 주권자들이 가장 많이 찾는 구체적인 단어의 형태를 예시로 보여줍니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 검색어를 입력해 주세요. | 질병명 또는 병원 이름 검색 |
| | 예) 청년 주택, 자동차세 연납 |

---

### 6.10 탭 (Tabs): 상호 배타적이고 완벽한 분류

탭의 이름표(레이블)는 서로 겹치지 않아야 하며, 탭을 눌렀을 때 나올 내용을 100% 예측할 수 있어야 합니다.

**규칙 1 — 상호 배타성 (MECE 원칙)**

탭의 분류 기준이 혼재되어 주권자가 "어느 탭을 눌러야 할지" 고민하게 만들면 안 됩니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 전체 / 진행 중 / 중요한 민원 (겹침 발생) | 전체 / 진행 중 / 처리 완료 (시간/상태 흐름에 따른 완벽한 분류) |

**규칙 2 — 글자 수와 품사의 통일**

여러 개의 탭이 나란히 있을 때는 각 탭의 품사(명사형)와 글자 수를 최대한 비슷하게 맞춰 시각적인 리듬을 줍니다.

---

### 6.11 드롭다운과 셀렉트 상자 (Dropdown & Select): 선택의 기준을 먼저 제시하라

클릭하기 전 '기본값(Default)' 텍스트가 힌트 역할을 해야 합니다.

**규칙 1 — 구체적인 기본값 (지시형 레이블)**

아무것도 선택하지 않은 초기 상태일 때, 단순히 "선택"이라고만 두지 말고 무엇을 선택해야 하는지 구체적으로 적어줍니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 선택 ∨ | 신청 연도 선택 ∨ |
| | 통신사 선택 ∨ |

**규칙 2 — 선택지의 품사 통일**

드롭다운 목록을 펼쳤을 때 나오는 항목들의 품사와 형태(글자 수 등)를 통일합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 단기 / 1년 이상 / 장기로 신청함 | 1년 미만 / 1년~3년 / 3년 이상 |

---

### 6.12 표와 데이터 그리드 (Data Table): 여백과 혼란을 통제하라

세금 내역, 민원 처리 현황 등 정부 서비스에서 절대 빠질 수 없는 것이 '표'입니다. 표 안의 텍스트는 극한의 '정보 다이어트'가 필요합니다.

**규칙 1 — 열(Column) 제목은 최단 명사로**

열의 제목(헤더)은 2~4글자의 아주 짧은 명사형으로 압축하여 공간을 확보합니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 신청하신 날짜 | 신청일 |
| 담당 기관명 | 담당 기관 |

**규칙 2 — 빈칸(Empty Cell)의 명시적 처리**

데이터가 없다고 해서 칸을 완전히 비워두면 주권자는 '시스템 오류'인지 '진짜 데이터가 없는 것'인지 헷갈립니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| (빈칸) | — (대시 기호) 또는 해당 없음 |

---

### 6.13 배너와 인라인 경고 (Banner & Inline Alert): 확성기의 볼륨 조절

화면 전체에 띄우는 글로벌 배너와 특정 영역에만 뜨는 인라인 알림은 공간이 좁은 만큼 핵심만 남겨야 합니다.

**규칙 1 — 배너는 '제목 + 액션'의 2단 구조**

배너에 구구절절 배경 설명을 넣으면 아무도 읽지 않습니다. 무슨 일인지(제목)와 어디로 가야 하는지(버튼/링크)만 남깁니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 2026년 귀속 연말정산 서비스가 1월 15일부터 시작되오니 많은 이용 바랍니다. | 2026년 연말정산 간소화 서비스 시작 → (클릭 시 이동) |

**규칙 2 — 인라인 경고의 시각적 결속**

입력 폼 등에서 발생하는 경고 문구는 반드시 문제가 발생한 그 **지점(Context) 바로 밑**에 붙여 써서 시선이 흩어지지 않게 합니다.

---

### 6.14 탐색 경로 (Breadcrumb): 헨젤과 그레텔의 빵 부스러기

현재 내가 어느 부처, 어느 메뉴의 몇 단계 깊이에 들어와 있는지 알려주는 네비게이션 텍스트입니다.

**규칙 — 페이지 타이틀과의 100% 일치**

탐색 경로에 쓰인 메뉴명은 실제 그 페이지의 H1(가장 큰 제목)과 글자 하나 다르지 않고 완벽하게 일치해야 합니다. 임의로 줄여 쓰면 다른 페이지로 착각하게 됩니다.

| 🚫 금지 | ✅ 권장 |
|--------|--------|
| 홈 > 복지 > 청년 지원 (실제 페이지 제목은 '청년 주거 지원금') | 홈 > 복지 > 청년 주거 지원금 |

---

> **6장 요약**: 버튼부터 탐색 경로까지, UI 컴포넌트의 텍스트는 모두 **행동 예측성**, **인지 마찰 제거**, **상태의 명확한 위계**라는 세 가지 렌즈로 점검합니다.

## 부록 C: 거버넌스 조건 — 일관성 유지를 위한 운영 프레임워크

> 이 부록은 KRDS UX Writing 가이드라인이 실제 조직에서 살아 숨 쉬는 문서로 유지되기 위한 운영 조건을 정의합니다. 각 조건은 **무엇을(조건)**, **어떻게(실행 방안)**, **누가(담당)**, **언제(갱신 주기)** 네 요소로 구성됩니다.

---

### C-1. 단일 진실 공급원 (Single Source of Truth)

#### C-1 조건

UX Writing 가이드라인은 오직 **하나의 마스터 문서**에서 관리한다. 복사본·별도 슬라이드·팀 위키 등 파생 문서가 마스터보다 우선되어서는 안 된다.

#### C-1 실행 방안

| 항목 | 내용 |
|------|------|
| 마스터 문서 위치 | GitHub Pages 배포본 (`principles.md` → `index.html`) |
| 참조 URL 고정 | 모든 내부 채널(Slack·Notion·Jira)에서 단일 URL 링크 사용 |
| 복사본 금지 원칙 | "내용 인용 시 URL 링크로 대체" — 전문(全文) 복붙 금지 |
| Notion 연동 방식 | Embed 방식으로 마스터 URL 임베드, Notion 내 독립 편집 금지 |
| 업데이트 알림 | 마스터 문서 갱신 시 #ux-writing Slack 채널에 변경 요약 공지 |

#### C-1 담당자

| 역할 | 책임 |
|------|------|
| UX Writer (주) | 마스터 문서 편집 권한 보유, 변경 이력 관리 |
| 서비스 기획자 (부) | 팀 내 파생 문서 존재 확인 후 URL 링크로 대체 요청 |

#### C-1 갱신 주기

- **정기**: 분기 1회 (3월·6월·9월·12월 첫째 주)
- **수시**: 법령 개정, 서비스 대규모 업데이트, 신규 컴포넌트 도입 시

#### C-1 위반 시 처리

> 파생 문서 발견 시: 발견자가 UX Writer에게 알림 → UX Writer가 해당 문서에 "이 문서는 구버전입니다. 최신 가이드: [링크]" 안내 추가 → 원본 삭제 또는 URL 대체 완료 후 종료

### C-3. Living Glossary 동기화

#### C-3 조건

금지어·대체어 사전(2.1 대체어 사전)은 서비스 변화에 따라 **지속적으로 업데이트**되어야 한다. 특히 시스템 변수(`%s`, `%i` 등)와 상태값 용어는 개발 부서 명칭과 항상 동기화한다.

#### C-3 실행 방안

| 항목 | 내용 |
|------|------|
| 사전 관리 플랫폼 | `principles.md` 2.1 섹션 (GitHub) + Notion 미러링 |
| 신규 용어 등록 프로세스 | 요청자 → UX Writer 검토 → 마스터 문서 반영 → 변경 알림 |
| 시스템 변수 동기화 | 개발팀과 분기 1회 용어 목록 대조 (API 응답값·상태코드 포함) |
| 상태값 명칭 기준 | 개발 코드 상태값과 화면 노출 텍스트 1:1 매핑 테이블 유지 |
| 폐기 용어 처리 | 취소선(~~취소~~)으로 표시 후 6개월 경과 시 삭제 |

**시스템 변수 표기 예시**

| 변수 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| `%s` (사용자명) | `%s님 안녕하세요` | `{이름}님, 반갑습니다` (테스트 시 실명으로 치환 확인 필수) |
| `%i` (숫자) | `%i건 처리` | `{건수}건이 처리됐습니다` |
| `%d` (날짜) | 날짜 형식 미지정 | `{날짜}(YYYY.MM.DD)까지 완료해 주세요` |

#### C-3 담당자

| 역할 | 책임 |
|------|------|
| UX Writer (주) | 사전 최종 편집 권한, 동기화 주관 |
| 개발팀 PL (협력) | 시스템 변수·상태값 목록 제공 (분기 1회) |
| 서비스 기획자 (협력) | 신규 서비스 기획 시 용어 사전 사전 검토 요청 |

#### C-3 갱신 주기

- **정기**: 분기 1회 (개발팀 동기화 미팅과 연계)
- **수시**: 신규 서비스 출시 전, 법령 개정으로 공식 용어 변경 시

### C-4. UX 라이팅 QA 의무화

#### C-4 조건

신규 서비스 런칭 또는 대규모 업데이트 시, 화면에 들어가는 모든 텍스트는 **UX Writing QA**를 거쳐야 한다. QA 없이 배포되는 텍스트는 없다.

#### C-4 실행 방안

**QA 의무 발동 기준**

| 상황 | QA 의무 |
|------|---------|
| 신규 서비스 런칭 | 전체 화면 텍스트 QA 필수 |
| 주요 플로우 변경 (로그인·신청·결제 등) | 변경된 플로우 전체 텍스트 QA 필수 |
| 버튼·레이블 10개 이상 수정 | 해당 화면 QA 필수 |
| 오류 메시지 신규 추가 | 3단 구조 QA 필수 (부록 B 섹션 3 참조) |
| 단순 오타 수정, 1–2개 텍스트 교체 | QA 생략 가능 (기획자 자체 검토 후 배포) |

**QA 프로세스**

```
기획 완료 → UX 라이팅 검토 요청 (Jira 또는 디자인 파일 코멘트)
         → UX Writer 또는 협의체 검토 (부록 B 체크리스트 적용)
         → 피드백 반영
         → 최종 승인 → 개발 전달
```

**최종 결정권 구조**

| 상황 | 결정권자 |
|------|---------|
| UX Writer 존재 시 | 전담 UX Writer |
| UX Writer 부재, 협의체 운영 시 | UX Writing 협의체 다수결 (기획·디자인·개발 각 1인 이상) |
| 긴급 배포 (24시간 이내) | 서비스 기획 리드 자체 결정 + 사후 UX Writer 리뷰 의무 |

**QA 예외 상황 처리 기준**

> **긴급 배포**로 QA를 생략한 경우:
> - 배포 후 5영업일 이내 사후 리뷰 진행
> - 개선 필요 사항 발견 시 다음 스프린트에 수정 티켓 생성

> **법적 필수 문구**로 원문 수정이 불가한 경우:
> - 법적 문구 아래 "쉽게 말하면:" 요약 섹션 추가 (변형 불가 원문 유지)

#### C-4 담당자

| 역할 | 책임 |
|------|------|
| UX Writer (주) | QA 수행, 최종 승인 |
| 서비스 기획자 (요청) | QA 요청, 일정 조율 |
| 디자이너 (지원) | 디자인 파일에 텍스트 QA 코멘트 스레드 생성 |

#### C-4 갱신 주기

- QA 체크리스트(부록 B)와 동기화: 부록 B 개정 시 QA 의무 기준도 함께 검토
- 연 1회 QA 예외 사례 회고를 통해 기준 정교화

### C-6. 철학 내재화 교육

#### C-6 조건

UX Writing 가이드라인은 읽는 것만으로는 내재화되지 않는다. **정기 교육**을 통해 실무자가 "왜 주권자 중심 언어를 써야 하는가"를 스스로 판단할 수 있도록 철학적 기반을 교육한다.

#### 교육 철학 핵심 메시지

> "정부 서비스의 언어는 권위를 드러내는 수단이 아닙니다.  
> 주권자가 자신의 권리를 행사하도록 돕는 도구입니다."  
> — 제1장 파운데이션에서

교육은 이 한 문장의 의미를 실무 맥락에서 되짚는 데 집중합니다.

#### 교육 커리큘럼 개요

**입문 과정** (신규 입사자·서비스 첫 담당자 대상, 2시간)

| 모듈 | 내용 | 시간 |
|------|------|------|
| 1. 왜 이 언어인가 | 제1장 파운데이션: 주권자 개념, 3대 원칙 배경 | 30분 |
| 2. 나쁜 글과 좋은 글 | Before/After 사례 10개 (행정어, 이중부정, 오류 메시지) | 40분 |
| 3. 직접 써보기 | 실습 워크시트: 실제 서비스 화면 3개 텍스트 수정 | 30분 |
| 4. 체크리스트 숙지 | 부록 B 자가진단 체크리스트 개요 | 20분 |

**심화 과정** (기획자·디자이너·시니어 개발자 대상, 2시간)

| 모듈 | 내용 | 시간 |
|------|------|------|
| 1. 오류 메시지 설계 | 4.1 에러 3단 구조 심화: 트리거별 실습 | 40분 |
| 2. 거버넌스 운영 | 이 부록 C 전체: SSoT·Glossary·QA·교육 내면화 | 30분 |
| 3. 케이스 스터디 | 실제 민원 서비스 3개 현황 진단 + 개선안 도출 | 50분 |

**연간 갱신 세션** (전 실무자 대상, 1시간)

| 내용 | 상세 |
|------|------|
| 가이드라인 변경 사항 공유 | 전년도 개정 내용 요약 + 사례 |
| 현장 Q&A | 실무 중 판단이 어려웠던 사례 공유 및 기준 명확화 |
| 체크리스트 업데이트 | 부록 B 신규 항목 공지 |

#### 실무자 자가 판단 기준 제공

> 교육 참여 여부와 무관하게, 아래 세 가지 질문으로 스스로 판단합니다.

**3-Question Self-Check**

```
1. "14세 중학생이 사전 없이 이해할 수 있는가?"
   → 아니라면 무번역 원칙 적용

2. "결론이 첫 문장에 있는가?"
   → 아니라면 정보핵심화 원칙 적용

3. "이 텍스트를 읽은 사람이 다음 행동을 알 수 있는가?"
   → 아니라면 심리적안전망 원칙 적용
```

이 세 질문을 통과하면 UX Writer의 사전 검토 없이도 배포 가능합니다.

#### C-6 담당자

| 역할 | 책임 |
|------|------|
| UX Writer (주) | 교육 커리큘럼 설계·진행, 워크시트 최신화 |
| HRD 또는 교육 담당자 (협력) | 교육 일정 편성, 입사자 온보딩 연계 |
| 팀 리드 (지원) | 팀원 교육 참여 독려, 갱신 세션 필수 참여 확인 |

#### C-6 갱신 주기

- **입문·심화 과정**: 가이드라인 주요 개정 시 커리큘럼 업데이트 (연 1–2회)
- **연간 갱신 세션**: 매년 1월 (전년도 회고 포함)
- **워크시트**: 서비스 사례 교체 주기 = 6개월 (현행 서비스 반영)
