## 요약 (Executive Summary)

- **KRDS는 컴포넌트·토큰·코드에 집중된 디자인 시스템이며, UX Writing 원칙은 거의 부재하다.** 버튼 레이블 동사형 원칙 1개가 유일한 텍스트 작성 기준이며, 에러 메시지·폼 레이블·안내문·접근성 텍스트의 작성 기준은 전혀 없다.
- **공식 원칙과 실제 UI 텍스트 사이에 자기 모순이 존재한다.** KRDS는 "동사형 레이블 원칙"을 명시하면서, 자사 사이트의 버튼에 "더보기", "초기화" 같은 명사형을 그대로 사용하고 있다.
- **에러 메시지에 심리적 부담을 주는 표현이 있다.** 검색 결과 없음 메시지에 "맞춤법 오류가 있는지 확인해 주세요"처럼 사용자의 실수를 암묵적으로 전제하는 언어가 사용된다. 타이포("검색어 에")도 공식 사이트에 노출 중이다.
- **사이트 전반에 톤앤매너 가이드가 적용되지 않아 문어체와 구어체가 혼재한다.** Hero 섹션의 "준비되어 있어요"와 컴포넌트 설명의 "가이드를 의미한다."가 같은 사이트에 공존한다.
- **우리 3대 원칙(무번역·정보핵심화·심리적안전망)은 KRDS 공식 원칙이 선언만 하고 실행 기준을 제공하지 못하는 빈 자리를 정확히 채운다.** "쉽게 이해할 수 있는 서비스"라는 원칙 선언과, 어떤 단어가 쉽고 어려운지를 판별하는 기준 제공은 전혀 다른 수준의 기여다.

## 1. 사이트 구조 및 네비게이션

### 1.1 GNB 전체 구조

KRDS 공식 사이트는 7개 1depth 카테고리, 총 42개 2depth 메뉴로 구성된다.

| 1depth | 2depth 수 | 주요 대상 |
|--------|-----------|---------|
| 시작하기 | 5 | 디자이너·개발자·정부관계자 온보딩 |
| 디자인 스타일 | 9 | 색상·타이포·아이콘·토큰 등 시각 기반 |
| 컴포넌트 | 11 카테고리 | UI 빌딩 블록 (46개 컴포넌트) |
| 기본 패턴 | 12 | 반복 과업 인터페이스 집합 |
| 서비스 패턴 | 5 | 사용자 여정 기반 전체 플로우 |
| 소식·소통 | 6 | 커뮤니티·공지·자료실 |
| KRDS 소개 | 7 | 원칙·정책·저작권 |

**메인 페이지 섹션 구성**:
1. **Hero** — "모두를 위한 디지털 서비스 경험" + CTA "시작하기"
2. **시작하기** — 디자이너/개발자/정부관계자 3탭 UI
3. **주요 구성** — 디자인 원칙·디지털 포용·스타일·컴포넌트·기본 패턴·서비스 패턴 카드 6개
4. **디자인 적용 사례** — 공정거래위원회·온통청년 사례
5. **의견 수렴** — "여러분의 소중한 의견을 기다립니다" + CTA "지금 참여하기"

### 1.2 정보 계층 분석

```
KRDS
├── 시작하기 (5) — 온보딩 전용
├── 디자인 스타일 (9) — 원자 디자인의 원자 계층
├── 컴포넌트 (11카테고리, 46개) — 분자·유기체 계층
├── 기본 패턴 (12) — 반복 과업 집합
├── 서비스 패턴 (5) — 사용자 여정 플로우
├── 소식·소통 (6) — 커뮤니티
└── KRDS 소개 (7) — 원칙·정책
```

**계층 구조 평가**:

| 항목 | 평가 | 근거 |
|------|------|------|
| 깊이 | 적절 (2~3depth) | 공공 사이트 기준 탐색 부담 낮음 |
| 논리 흐름 | 스타일→컴포넌트→패턴 순서 | 원자 설계(Atomic Design) 철학과 일치 |
| 중복 진입점 | 문제 있음 | "디자인 원칙"이 메인 카드와 KRDS소개 양쪽에 등장 |
| KRDS소개 vs 시작하기 | 혼재 위험 | 디자인 원칙·네이밍 원칙이 "소개" 섹션에 격리되어 학습 흐름 단절 |

**대상 사용자별 멘탈 모델 일치도**:

| 사용자 | 기대 태스크 | KRDS 대응 | 일치도 |
|--------|-----------|-----------|-------|
| 디자이너 | Figma 리소스, 토큰, 컴포넌트 스펙 | 튜토리얼·리소스·토큰 탭 | 높음 |
| 개발자 | 마크업 가이드, CSS 토큰, 코드 예시 | 컴포넌트 "코드" 탭 | 높음 |
| 정부 관계자 | 적용 사례, 의무 기준, 감사 대비 | 소개·적용 사례 | 보통 (감사 기준 불명확) |
| 일반 시민 | — | 직접 대상 콘텐츠 없음 | 해당 없음 |

### 1.3 주요 섹션별 내용 요약

- **시작하기**: 3유형 탭 UI로 디자이너(튜토리얼·리소스·토큰), 개발자(마크업·코드), 정부관계자(소개·사례) 분기
- **컴포넌트**: 개요/접근성/코드 3탭 구조. 각 페이지 정의→구조→용례(Do/Don't)→사용성 가이드라인→변경이력 순
- **서비스 패턴**: 방문·검색(8단계)·로그인(8단계)·신청(8단계)·정책 정보 확인(4단계). Do/Better/Best 3단계 적용 기준 제시
- **소식·소통**: 새소식·FAQ·적용 사례·자료실·문의·동영상
- **KRDS 소개**: 7대 디자인 원칙, 5대 네이밍 원칙, 디지털 포용, 저작권

## 2. 디자인 시스템 구성요소

### 2.1 컴포넌트 카테고리 (11개 카테고리, 46개 컴포넌트)

| 카테고리 | 컴포넌트 수 | 주요 컴포넌트 |
|---------|-----------|------------|
| 아이덴티티 | 4 | 공식 배너(Masthead), 운영기관 식별자, 헤더, 푸터 |
| 탐색 | 6 | 건너뛰기 링크, 메인 메뉴, 브레드크럼, 사이드 메뉴, 콘텐츠 내 탐색, 페이지네이션 |
| 레이아웃 및 표현 | 13 | 구조화 목록, 긴급 공지, 달력, 디스클로저, 모달, 배지, 아코디언, 이미지, 캐러셀, 탭, 표, 텍스트 목록, 파비콘 |
| 액션 | 3 | 링크, 버튼, 플로팅 액션 버튼(FAB) |
| 선택 | 5 | 라디오 버튼, 체크박스, 셀렉트, 태그, 토글 스위치 |
| 피드백 | 2 | 단계 표시기(Step indicator), 스피너 |
| 도움 | 6 | 도움 패널, 따라하기 패널, 맥락적 도움말, 코치마크, 툴팁, 음성지원(TTS) |
| 입력 | 4 | 날짜 입력 필드, 텍스트 영역, 텍스트 입력 필드, 파일 업로드 |
| 설정 | 2 | 언어 변경, 화면 크기 조정 |
| 콘텐츠 | 2 | 접근 가능한 미디어, 숨긴 콘텐츠(Visually hidden) |
| 모바일 전용 | 8 | 바텀시트, 토스트, 스낵바, 탭바, 스플래시 스크린, 수량토글, 범위 슬라이더, 뒤로가기 |

**도움(Help) 카테고리가 6개로 다른 디자인 시스템 대비 매우 풍부하다.** 이는 공공 서비스 사용자의 디지털 리터러시 편차가 크기 때문으로 해석된다. 도움 패널·따라하기 패널·코치마크처럼 사용자가 모르더라도 안내받을 수 있는 컴포넌트에 투자한 것은 "모든 사용자를 포용"하는 원칙의 실질적 구현이다.

### 2.2 서비스 패턴 (5개)

| 패턴 | 단계 수 | 특이사항 |
|------|--------|---------|
| 방문 | 1 | — |
| 검색 | 8 | 개요→검색 기능 찾기→검색어 입력→결과 확인→상세 검색→결과 이용→재검색→종료 |
| 로그인 | 8 | 다중 인증 방식 선택 단계 별도 포함 |
| 신청 | 8 | 유의사항/자격 확인 단계 분리 — 오류 제출 방지 |
| 정책 정보 확인 | 4 | 정책 탐색→정보 확인→자료 탐색 |

서비스 패턴마다 **Do(필수)·Better(권장)·Best(우수)** 3단계 적용 수준을 제시한다. 이는 공공기관 감사 대응 및 예산·역량이 다른 기관들의 현실적 적용을 고려한 설계다.

### 2.3 디자인 원칙 (7개)

| # | 원칙명 | 핵심 내용 |
|---|--------|---------|
| 1 | 사용자 중심 | 사용자 요구 기반 설계, 목적 달성 지원 |
| 2 | 모든 사용자를 포용 | 장애인·고령자·어린이·외국인 포함한 전체 시민 |
| 3 | 공통+개별 | 공통 경험 기반 위에 개별 맥락 반영 |
| 4 | 빠르고 간단 | 의사결정 최소화, 단순한 경험 설계 |
| 5 | 쉽게 이해·사용 | 친숙·간결·직관적, 도움 없이 이용 가능 |
| 6 | 다양한 상황 고려 | 숙련도·이용빈도·이용 상황 배려 |
| 7 | 신뢰 | 공식 서비스 인지 + 최신·정확 정보 접근 |

### 2.4 네이밍 원칙

KRDS 네이밍 원칙은 디자인 토큰 명명 기준이나, UX Writing에도 직접 적용 가능한 원칙이다.

```
네이밍 구조: namespace > theme > category > component > type > modifier
예시: krds > light > color > button > background > primary
```

| 원칙 | 내용 | UX Writing 적용 |
|------|------|---------------|
| 일관성 | 논리 구조에 따른 명명 | 동일 개념 동일 표현 |
| 기능 기반 | 시각 속성 배제, 역할 기반 | 레이블은 기능을 말해야 함 |
| 확장성 | primary/secondary 구분 | 버튼 위계 레이블 체계 |
| 통일 표기 | 하이픈(-) 구분자 통일 | 표기법 통일 |
| 명확성 | 줄임말 지양 ("bg" → "background") | 약어 금지 |

## 3. UX Writing 관점 분석

### 3.1 언어 수준 평가

**행정 관습어 현황**:

| 발견 텍스트 | 유형 | 대중 친화성 |
|------------|------|-----------|
| "누리집" | 행정 조어 (인터넷 사이트 공식 대체어) | 낮음 — 14세 기준 이해 어려움 |
| "전자정부 누리집" | 행정 복합어 | 낮음 — "공식 정부 웹사이트"로 풀어야 이해 가능 |
| "행정안전부 산하기관" | 행정 위계 표현 | 보통 — 관계 파악에 배경 지식 필요 |
| "웹접근성 인증" (기관명 포함) | 전문 기관명 나열 | 낮음 — 기관명이 과도하게 길고 낯섦 |

**컴포넌트 네이밍 한글화 품질**:

| 컴포넌트명 | 평가 | 개선 방향 |
|-----------|------|---------|
| 건너뛰기 링크 (Skip link) | 적절 | — |
| 디스클로저 (Disclosure) | 미흡 | "접기·펼치기"가 더 직관적 |
| 엘리베이션 (Elevation) | 미흡 | "시각적 깊이·그림자 효과"로 설명 필요 |
| 코치마크 (Coach mark) | 보통 | "초보자 안내 표시"로 대체 가능 |
| 플로팅 (FAB) | 미흡 | "플로팅"은 행동 의미 전달 약함 |

**14세 기준 이해 가능성 평가**:

| 영역 | 이해도 | 문제점 |
|------|--------|--------|
| Hero 카피 | 중상 | "경험을 누린다" 표현이 다소 어색 |
| 시작하기 탭 설명 | 중 | "디자인 시스템 구축"은 IT 전문 독자 대상어 |
| 컴포넌트·패턴 설명 | 하 (전문가 대상) | 전문가 대상이므로 적절하나, 정부관계자 탭에서도 동일 수준이면 문제 |
| 서비스 패턴 Do/Better/Best | 중 | 영어 키워드 혼용이 어색. 한글 레이블 단독 사용 권장 |
| 버튼 레이블 | 상 | 단순 동사형, 이해 용이 |
| 푸터 안내 | 중하 | "누리집" + 소관기관 나열이 내부 조직 지향적 |

### 3.2 톤앤매너 분석

**이중 페르소나 문제**: KRDS 사이트는 두 가지 다른 독자(전문가·시민)를 동시에 대상으로 하면서 톤이 혼재한다.

| 섹션 | 독자 | 톤 | 종결 어미 |
|------|------|-----|---------|
| Hero / 메인 CTA | 시민+관계자 | 친근 시도, 포용적 | ~갑니다 |
| 시작하기 탭 | 전문가 | 실용적, 구어 시도 | ~있어요, ~보세요 |
| 주요 구성 카드 | 전문가 | 정의 중심, 건조함 | ~이다 |
| 의견 수렴 | 전체 | 감성적, 포용적 | ~들려주세요 |
| 푸터 | 공식 | 행정 언어 | ~입니다, ~이다 |
| 문의 섹션 | 전체 | 최고 격식 | ~있으십니까? |

**경어·존댓말 불일치 사례**:

```
구어체: "디자인 시스템 구축을 위한 모든 자료가 준비되어 있어요.
         지금 바로 최신 정보를 확인하고, 프로젝트를 시작해 보세요."

문어체: "사용자 인터페이스의 가장 작은 단위로 일관성 있게 사용되는
         공통 요소에 대한 가이드를 의미한다."
```

같은 사이트 내에서 `~있어요` (구어)와 `~의미한다` (문어 종결)가 공존한다. 톤 가이드가 없거나 적용이 일관되지 않은 전형적 사례다.

### 3.3 UI 텍스트 패턴별 분석

#### 버튼 레이블

**긍정 사례**:
- `시작하기` — 동사형, 결과 예측 가능
- `지금 참여하기` — 동사형 + 시간 부사로 긴급성 부여
- `코드 확인하기` / `코드 복사하기` — 목적어 포함 동사형, 명확함

**개선 필요 사례**:

| 현재 텍스트 | 문제점 | 개선 제안 |
|------------|--------|---------|
| `더보기` / `자세히 보기` | 맥락 없음 — 무엇을 더 보는지 불명확 | `사례 더보기`, `버튼 상세 보기` |
| `파일다운로드` | 띄어쓰기 오류, 목적어 없음 | `리소스 파일 다운로드` |
| `초기화` | 명사형, 대상 불명확 | `검색 조건 지우기` |
| `열기` | 대상 없음 | `메뉴 열기`, `파일 열기` |
| `찜하기` | 공공기관 서비스 어조 부적합 여부 검토 필요 | — |

#### 에러 메시지

**검색 결과 없음 메시지 현황**:

```
현재:
"검색어 에 대한 검색 결과가 없습니다."  ← 공백 오류
"철자나 맞춤법 오류가 있는지 확인해 주세요"
"여러 단어로 나누거나 다른 검색어로 검색해 주세요."
"보다 일반적인 단어로 검색해 주세요."
```

문제점:
1. **타이포**: `검색어 에` — 공백 오류가 공식 사이트에 노출
2. **심리적 부담**: "철자나 맞춤법 오류가 있는지"는 사용자 실수를 암묵적으로 전제
3. **공감 없음**: 공감 표현 없이 해결책만 나열
4. **원인 미설명**: 왜 결과가 없는지 설명 없이 해결책 목록 제시

개선 방향:
```
"'{검색어}'에 대한 결과를 찾지 못했습니다.
아래 방법을 시도해 보세요:
• 단어를 짧게 줄여보세요 (예: '행정 절차 안내' → '절차')
• 비슷한 표현으로 다시 검색해 보세요"
```

#### 안내문·설명문

현재 패턴은 **배경→결론** 구조로, 역피라미드(결론 우선)가 적용되지 않는다.

| 현재 텍스트 | 문제 | 개선 방향 |
|------------|------|---------|
| "사용자 인터페이스의 가장 작은 단위로 과업에 상관없이 일관성 있게 사용되는 공통 요소에 대한 가이드를 의미한다." | 핵심이 문장 끝에 위치, 수식어 과다 | "UI를 만드는 기본 재료입니다. 버튼·입력창처럼 어디서나 같은 모양으로 쓰는 요소를 다룹니다." |
| "디지털 정부서비스에서 제공하는 핵심 과업에 대한 사용자 여정 기반의 사용자 경험 설계 가이드를 의미한다." | "사용자" 2회 반복, 과도한 명사 체인 | "민원 신청·검색·로그인처럼 자주 하는 작업을 어떻게 설계할지 단계별로 안내합니다." |
| "국민 누구나 디지털 정부서비스 경험을 쉽게 누릴 수 있는 기준을 만들어갑니다." | 상대적으로 양호. "경험을 누린다" 어색 | "국민 누구나 디지털 정부서비스를 쉽게 쓸 수 있는 기준을 만듭니다." |

#### 플레이스홀더

| 항목 | 현재 텍스트 | 문제 | 개선 제안 |
|------|------------|------|---------|
| 검색창 | "찾고자 하는 메뉴명을 입력해 주세요." | "메뉴명"으로 검색 범위를 좁게 암시. 실제로는 컴포넌트·원칙명도 검색 가능 | "찾고 싶은 내용을 검색하세요" |

### 3.4 발견된 문제점 (자기 모순·톤 혼재 등 구체적 사례)

**문제 1 — 자기 모순 (가장 심각)**

KRDS는 버튼 레이블 가이드에서 명확히 선언한다:

> "버튼의 텍스트 레이블은 버튼을 통해 실행되는 기능을 예측할 수 있는 명확한 내용을 동사형으로 제공한다."

그러나 자사 사이트에는:
- `더보기` (명사형, 동사형 원칙 위반)
- `초기화` (명사형, 대상 불명확)
- `파일다운로드` (명사형, 띄어쓰기 오류)

"원칙을 문서화하되 스스로 지키지 않는" 자기 모순은 가이드라인 신뢰도에 직접적 타격을 준다.

**문제 2 — 심리적 안전망 부재**

검색 결과 없음 메시지:
> "철자나 맞춤법 오류가 있는지 확인해 주세요"

이 표현은 사용자의 잘못을 암묵적으로 전제한다. 실제로는 검색 인덱스 부재, 다른 명칭 사용 등 시스템 측 원인일 수 있다. "심리적안전망" 원칙 — 실패 시 사용자를 탓하지 않는 언어 — 위반.

**문제 3 — 톤 혼재**

| 위치 | 텍스트 | 종결 어미 |
|------|--------|---------|
| 시작하기 탭 | "준비되어 있어요" | ~요 (구어) |
| 컴포넌트 소개 | "가이드를 의미한다" | ~다 (문어) |
| 문의 섹션 | "의견이 있으십니까?" | ~까? (최고 격식) |

동일 사이트에서 3가지 다른 종결 어미 체계가 공존한다.

**문제 4 — 타이포그래피 오류**

> "검색어 에 대한 검색 결과가 없습니다."

`검색어`와 `에` 사이에 불필요한 공백이 있다. 공공 기관의 공식 사이트에서 맞춤법·문자 오류는 신뢰도를 직접 훼손한다.

## 4. KRDS 공식 원칙 vs 우리 3대 원칙 비교

| 항목 | KRDS 공식 | 우리 원칙 | 평가 |
|------|-----------|-----------|------|
| **무번역(No-Translation)** | 원칙 5 "쉽게 이해·사용" — "친숙하고 직관적인 방식" 선언 | 행정어·전문용어를 번역 없이 이해 가능한 단어로 교체하는 구체적 실행 기준 | KRDS는 선언만 함. 우리는 어떤 단어가 쉽고 어려운지 판별 기준을 제공 → **확장** |
| **정보핵심화(Essentialism)** | 원칙 4 "빠르고 간단" — "의사결정 최소화" 선언 | 역피라미드(결론 우선), 필수 정보만, 단어 수 제한 실행 기준 | KRDS는 원칙 선언만. 실제 텍스트는 배경→결론 구조. 우리가 정보 구조 원칙 실행 도구를 제공 → **신규 공백** |
| **심리적안전망(Psychological Safety)** | 원칙 7 "신뢰" — 정보의 정확성·공식성 신뢰 중심 | 실패 시 사용자를 탓하지 않는 언어, 에러 메시지 3단 구조 | KRDS의 "신뢰"는 기관 신뢰(공식성·정확성). 우리의 심리적안전망은 사용자 정서 신뢰(실패해도 괜찮음). 방향이 다름 → **신규 영역** |
| 포용성 | 원칙 2 "모든 사용자를 포용" — 장애인·고령자·외국인 포함 | 인지 다양성(읽기 어려움) 포함 | KRDS는 물리적 접근성 중심. 우리는 인지 접근성(언어 수준)으로 확장 → **확장** |
| 일관성 | 네이밍 원칙 1 "일관성" — 토큰 명명 체계 | 동일 개념 동일 표현, 섹션 간 톤 일관성 | KRDS는 토큰 레벨 일관성. 우리는 텍스트 레벨 일관성으로 적용 범위 확장 → **확장** |
| 동사형 레이블 | 버튼 가이드 "동사형 원칙" 명시 | 모든 UI 텍스트로 원칙 확장 | KRDS는 버튼 레이블에만 적용. 우리는 폼 레이블·플레이스홀더·헤딩까지 확장 → **확장** |
| 에러 메시지 언어 | 오류 기본 패턴(global_07) 존재하나 텍스트 기준 없음 | 에러 메시지 3단 구조, 금지 표현 목록 | KRDS는 UI 구조만, 텍스트 원칙 전무 → **신규 공백** |

**결론**: 우리 원칙은 KRDS가 선언 수준으로 두고 있는 것들을 실행 수준으로 내려준다. 이는 중복이 아닌 레이어 심화(depth extension)다.

## 5. 재설계 인사이트

### 5.1 KRDS에서 흡수할 것

**흡수 1 — 서비스 패턴의 단계 구조**

KRDS 서비스 패턴(검색 8단계·로그인 8단계·신청 8단계)을 UX Writing 상황 맵핑 구조로 재활용할 수 있다.

예: "신청 4단계(신청서 작성)에서 어떤 텍스트를 써야 하는가"를 직접 찾을 수 있는 구조가 가능하다. 이는 정부 관계자가 즉시 적용 가능한 실용적 가이드를 제공한다.

**흡수 2 — Do/Better/Best 적용 수준 체계**

KRDS의 3단계 적용 기준(필수·권장·우수)은 예산과 역량이 다른 기관들의 현실을 반영한 실용적 설계다. 동일 체계를 UX Writing에 적용:
- **Do**: 최소 준수 기준 (에러 메시지에 사용자 탓 표현 금지)
- **Better**: 권장 (공감 → 대안 순서로 에러 메시지 작성)
- **Best**: 우수 (구체적 예시·다음 단계 포함 에러 메시지)

**흡수 3 — 버튼 레이블 동사형 원칙**

KRDS가 이미 문서화한 버튼 레이블 원칙을 출발점으로, 전체 UI 텍스트로 확장한다. 기존 원칙과 충돌하지 않으면서 우리 가이드의 신뢰 기반을 확보한다.

**흡수 4 — 네이밍 원칙 5가지**

일관성·기능 기반·명확성·줄임말 지양 원칙은 레이블·헤딩·버튼 텍스트 작성에 그대로 적용 가능하다.

### 5.2 KRDS가 다루지 않는 공백 영역 (우리가 채울 것)

| 공백 영역 | 설명 | 관련 원칙 |
|---------|------|---------|
| **에러·실패 언어 원칙** | 오류 기본 패턴(global_07)은 UI 구조만 다룸. 텍스트 작성 기준 전무 | 심리적안전망 |
| **폼 레이블·플레이스홀더 작성 원칙** | 입력 폼 패턴은 있으나 레이블 텍스트 기준 없음 | 무번역, 정보핵심화 |
| **안내문·도움말 톤앤매너** | 도움 컴포넌트 6개 있으나 그 안에 들어갈 텍스트 기준 없음 | 무번역, 심리적안전망 |
| **단어 난이도 기준표** | "쉽게 이해"를 원칙으로 선언하지만 어떤 단어가 쉬운지 기준 없음 | 무번역 |
| **정보 구조 원칙** | 역피라미드·결론 우선·문장 길이 기준 전무 | 정보핵심화 |
| **성공·완료 메시지 패턴** | "완료" 단계는 있으나 완료 화면 텍스트 기준 없음 | 심리적안전망 |
| **빈 상태·검색 없음 텍스트** | 단 1개 샘플 존재, 원칙 없음 | 심리적안전망 |
| **상황별 톤 가이드** | 사이트 내 톤 혼재가 톤 가이드 부재의 직접 증거 | 전체 |
| **알림·토스트 메시지** | 토스트·스낵바 컴포넌트 있으나 텍스트 기준 없음 | 정보핵심화, 심리적안전망 |
| **접근성 텍스트(alt·aria-label)** | 접근성 탭 있으나 대체 텍스트 작성 기준 없음 | 무번역, 포용 |
| **모바일 특화 Writing** | 모바일 컴포넌트 섹션 있으나 화면 폭 제한·터치 피드백 텍스트 기준 없음 | 정보핵심화 |

### 5.3 즉시 적용 가능한 액션 아이템

#### P1 (즉시) — 기반 원칙 수립

- **에러 메시지 3단 구조 정의**: 무슨 일이 있었는지 → 왜인지(사용자 탓 없이) → 다음에 할 수 있는 것
- **금지 표현 목록 제정**: "잘못된", "오류가 발생했습니다", "다시 시도해 주세요"(원인 없는 재시도), "맞춤법 오류가 있는지"
- **단어 난이도 기준표**: 행정어·전문어 20~30개 핵심 대체어 사전 (예: "누리집" → "웹사이트", "핵심 과업" → "주요 작업")
- **접근성 텍스트(alt·aria-label) 작성 원칙**: 기능 설명 우선, 장식 이미지 `alt=""`

#### P2 (단기) — 컴포넌트별 Writing 기준

- **버튼 레이블 확장 원칙**: KRDS 동사형 원칙 + 6자 이내 권장 + 맥락 포함 ("더보기" → "사례 더보기")
- **폼 레이블·플레이스홀더 기준**: 레이블은 명사형, 플레이스홀더는 범위를 좁히지 않는 표현
- **안내문 결론 우선 구조**: 1문장 요약 → 필요 시 상세 (역피라미드 적용)
- **성공·완료 메시지 패턴**: 완료 사실 확인 + 다음 단계 안내 + 긍정 강화
- **빈 상태 텍스트 기준**: 공감 → 대안 2~3개 제시 순서

#### P3 (중기) — 서비스 플로우 통합

- **KRDS 서비스 패턴 연동 Writing 가이드**: 신청·로그인·검색 각 단계별 적합한 텍스트 패턴 매핑
- **상황별 톤 가이드 제정**: 안내(중립·명확) / 에러(공감·회복) / 완료(따뜻·확인) / 경고(명확·즉각)
- **하십쇼체·합쇼체 금지 권고**: 디지털 전환 맥락에서 과도한 격식 체계 지양
- **알림·토스트 메시지 원칙**: 7자 이내 핵심만, 행동 유도 포함

```
  → 목적어 포함 동사형. 무엇을 어떻게 하는지 명확.

검색창 플레이스홀더: "찾고자 하는 메뉴명을 입력해 주세요."
  → 청유형, 정중함. 단, 범위를 "메뉴명"으로 좁히는 것은 문제.
```

### 개선이 필요한 사례

```
에러: "검색어 에 대한 검색 결과가 없습니다."
  → 공백 오류. 심리적 안전망 부재.

버튼: "더보기", "초기화", "파일다운로드"
  → 자사 원칙(동사형) 위반. 자기 모순.

안내문: "사용자 인터페이스의 가장 작은 단위로 과업에 상관없이 일관성 있게
         사용되는 공통 요소에 대한 가이드를 의미한다."
  → 수식어가 핵심을 뒤로 밀어냄. 역피라미드 미적용.

서비스 패턴 설명: "사용자 여정 기반의 사용자 경험 설계 가이드"
  → "사용자" 2회 반복. 명사 체인 과도.
```

### 컴포넌트 용어 이중 표기 샘플

```
건너뛰기 링크 (Skip link)  ← 한글명 직관적, 적절
디스클로저 (Disclosure)    ← 한글명이 의미를 담지 못함
엘리베이션 (Elevation)     ← 한글화 미흡
코치마크 (Coach mark)      ← "초보자 안내 표시"가 더 직관적
플로팅 (FAB)               ← 행동 의미 전달 약함
스낵바 (Snackbar)          ← 토스트와 혼용, 차이 설명 필요
```

### 서비스 패턴 적용 기준 표현 샘플

```
Do (필수)   — 반드시 구현
Better (권장) — 사용성 향상을 위한 권장
Best (우수)  — 최상 수준의 사용자 경험
```

한글과 영어가 혼용된다. `필수(Do)`, `권장(Better)`, `우수(Best)` 형태보다 `필수`, `권장`, `우수` 한글 단독 사용이 권장된다.
