## 1. 기관 개요 및 서비스 맥락

홈택스는 국세청이 운영하는 세금 신고·납부·증명 발급 통합 플랫폼입니다.  
과업 복잡도가 높고 실수 시 **가산세 등 실질적 불이익**이 발생하는 고위험 서비스입니다.  
사용자 감정 부하가 크며, 불복·체납·사망자 조회처럼 **심리적 민감 상황**이 빈번히 발생합니다.

| 서비스 유형 | 대표 과업 |
|------------|---------|
| 세금 신고 | 종합소득세, 부가가치세, 원천세 |
| 세금 납부 | 전자납부, 분납, 세금계산서 조회 |
| 증명서 발급 | 납세증명서, 소득금액증명, 사업자등록증명 |
| 환급 | 환급 신청, 환급 계좌 등록 |
| 전자세금계산서 | 발급, 조회, 수정 |
| 상담·불복 | 고충민원, 심판청구, 이의신청 |
| 장려금 | 근로장려금, 자녀장려금 신청 |

**분석 근거**:  
이 가이드는 홈택스 GNB(상단 내비게이션) 8개 대분류와 각 하위 메뉴 전체를 대상으로,  
KRDS UX Writing 3대 원칙(무번역·정보핵심화·심리적안전망)을 기준으로 실사 분석한 결과입니다.

홈택스 GNB 8개 대분류와 주요 하위 메뉴의 레이블을 사용자 관점에서 진단합니다.

### 2.0 GNB 전체 대분류 구조 (실사 확인)

홈택스 GNB 8개 대분류 (2026-04-24 기준):

```
1. 계산서·영수증·카드
2. 증명·등록·신청·사업장현황
3. 세금신고
4. 납부·고지·환급
5. 지급명세·자료·공익법인

7. 상담·불복·제보
8. 세무대리·납세관리
```

### 2.1 GNB 대분류 레이블 문제

| 현재 GNB 레이블 | 문제 유형 | 권고 방향 |
|----------------|---------|---------|
| 세금신고 | 복합 행정어 (신고 유형 불명시) | "신고하기" 또는 섹션 내 유형별 분기로 보완 |
| 납부·고지·환급 | 가운뎃점(·) 사용 — 읽기 흐름 끊김 | 내부 하위 구조 정비 후 "납부·환급" 축약 검토 |
| 상담·불복·제보 | 3항목 병렬 — "불복" 의미 모호 | "문의·이의신청" 또는 용어 설명 제공 |
| 장려금·연말정산·기부금 | 성격이 다른 3가지 혼합 (장려금=혜택, 연말정산=의무, 기부금=공제) | 사용자 역할 기반 분기 고려 |
| 세금 관련 신청 | "관련"의 범위 모호 | "신청·등록" 또는 과업 명칭으로 |
| 증명·발급 | 한국어 구조상 "증명서 발급"이 자연스러움 | "증명서 발급" |

### 2.1-A GNB 표기 3중 불일치 (★★★ 심각)

**같은 메뉴명이 한 페이지 내에서 3가지 다른 문자로 표기됩니다.**

| 위치 | 표기 | 사용 문자 |
|------|------|---------|
| GNB 탭 | `납부·고지·환급` | U+00B7 가운뎃점 (중간점) |
| 페이지 h1 제목 | `납부ㆍ고지ㆍ환급` | U+318D 한국어 가운뎃점 (반각) |
| 브레드크럼 | `납부 · 고지 · 환급` | 공백 + 가운뎃점 + 공백 |

이 세 표기는 시각적으로 유사하나 서로 다른 유니코드 문자입니다.
스크린리더와 검색 엔진이 다른 문자열로 처리하며, 복사 시 불일치 발생.

**권고**: 전 페이지에서 `·` (U+00B7, 공백 없음)으로 통일. 상세 기준은 6장 참조.

### 2.2 주요 하위 메뉴명 문제

| 하위 메뉴 현재 표현 | 문제 | 권고 표현 |
|------------------|------|---------|
| 종합소득세신고·지방소득세신고 | 두 과업 병합, 점 없이 붙여쓰기 | "종합소득세 신고" (지방소득세는 다음 단계에서 안내) |

| 세금계산서합계표 | 7자 복합 명사 | "세금계산서 합계표" |
| 원천징수이행상황신고서 | 9자 행정어 복합 명사 | "원천징수 신고서" (이행상황 제거) |

## 3. 섹션별 UX Writing 진단

### 3.1 공지사항

**위치**: 홈 > 공지사항, GNB > [공지사항 탭]

#### 발견 이슈

**이슈 1: `구분Hidden` 칼럼 — 스크린리더 노출**

공지사항 테이블의 첫 번째 칼럼 헤더가 `구분Hidden`으로 HTML 상에 존재하며,
스크린리더 사용자에게 "구분Hidden"이라는 텍스트가 그대로 읽힙니다.

- **유형**: 접근성 (WCAG 2.1 1.3.1 정보와 관계)
- **영향**: 시각 장애 사용자가 "구분Hidden"을 실제 컬럼명으로 인식
- **권고**: `aria-hidden="true"` 또는 `<th scope="col" class="sr-only">유형</th>` 처리
  시각 표시 없이 스크린리더에만 "유형"을 전달하거나, 칼럼 자체 숨김 처리

**이슈 2: 스크린리더용 캡션 텍스트 — 문법 오류 + 화면 노출**

테이블의 `<caption>` 텍스트가 스크린리더에 읽히는 것 자체는 정상이나,
내용이 개발자 내부 텍스트 형식이며 한국어 문법도 틀린 상태로 노출됩니다.

```
현재 캡션:
"공지사항 목록은(는) 번호,제목,작성일,조회수 을(를) 나타낸 표"

문제:
- "은(는)", "을(를)" — 서버 사이드 조사 선택 로직 실패 (은/를이 그대로 출력)
- 쉼표 뒤 공백 없음: "번호,제목,작성일" (→ "번호, 제목, 작성일")
- 화면에도 노출됨 (CSS 미처리 구간)
```

- **권고**: `aria-label="공지사항 목록 (총 {N}건)"` 또는 `<caption class="sr-only">공지사항 목록 (총 {N}건)</caption>`. 조사 처리 로직 수정.

**이슈 2-B: `첨부파일IDHidden` 테이블 헤더 노출**

`구분Hidden` 외에도 `첨부파일IDHidden` 컬럼 헤더가 테이블에 그대로 표시됩니다.

- **영향**: 스크린리더 사용자가 "첨부파일IDHidden"을 컬럼명으로 읽음
- **권고**: `aria-hidden="true"` 처리. 첨부파일 유무는 아이콘 + `aria-label`로 전달.

**이슈 2-C: `해당없음` 전 행 반복**

"구분" 컬럼의 모든 행에 "해당없음"이 표기됩니다.

- **영향**: 스크린리더 사용자가 모든 행에서 "해당없음"을 반복 청취
- **권고**: "구분" 컬럼 자체를 숨기거나, 구분이 있는 행에만 값 표시 (없으면 빈 칸)

**이슈 2-D: 페이지네이션 표기 비직관적**

| 현재 | 문제 | 권고 |
|------|------|------|
| `총108건(1/11)` | 공백 없음, 괄호 의미 불명확 | `전체 108건 \| 1 / 11 페이지` |
| 검색 버튼: `조회` | 맥락 없음 — 무엇을 조회? | `검색` 또는 `공지 검색` |

**이슈 3: "COMING SOON" 영문 레이블**

서비스 준비 중 상태를 "COMING SOON" 영문으로 표기. 한국어 서비스 일관성 위반.

```
실제 발견: "종합소득세 원클릭 환급 서비스 COMING SOON"
```

| 현재 | 권고 |
|------|------|
| `COMING SOON` | `(오픈 예정)` 또는 `준비 중` |
| `NEW` | `새 글` 또는 `N` 뱃지 |
| `HOT` | 삭제 (주관적 표현) |

### 3.2 세금신고

**위치**: GNB > 세금신고 > [14개 하위 항목]

#### 발견 이슈

**이슈 4: 14개 세금 종류 비분류 나열**

14개 세금 신고 종류가 카테고리화 없이 나열되어 있습니다.  
사용자는 "내가 해야 할 신고가 무엇인지" 찾는 맥락인데, 현재는 세금 명칭 목록을 스캔해야 합니다.

```
현재 구조 (나열식):
- 종합소득세신고·지방소득세신고
- 부가가치세신고
- 원천세신고
- 법인세신고
- 증여세신고
- 상속세신고
- 종합부동산세신고
- 개별소비세신고
... (14개)
```

**권고 방향**:  
사용자 역할(나는 누구인가) 기반 분기를 메뉴 진입 전에 제공.

```
✅ 권고 구조:
[직장인·프리랜서]  [개인사업자]  [법인사업자]  [세무 대리인]
```

**이슈 5: "신고내용확인 및 진행상황조회" 복합 메뉴명**

9어 복합 명사. 실제 사용자 니즈는 "내 신고가 잘 됐는지 확인"입니다.

| 현재 | 권고 |
|------|------|
| 신고내용확인 및 진행상황조회 | 신고 현황 조회 |
| 세금계산서합계표 | 세금계산서 합계표 |
| 원천징수이행상황신고서 | 원천징수 신고서 |

**이슈 6: 신고 유형 선택 화면의 복합 메뉴명**

종합소득세 하위 진입 후 "단순경비율·기준경비율 추계신고" 같은 세무사 전용 용어가 일반 납세자에게 노출.

**권고**: 사용자 유형별 화면 분기 또는 인라인 도움말(ⓘ) 제공

**이슈 6-B: 기능 혼합 메뉴명 (`신고서 조회/삭제/부속서류`)**

하나의 메뉴에 성격이 다른 3개 기능이 슬래시로 묶임.

| 현재 | 문제 | 권고 |
|------|------|------|
| `신고서 조회/삭제/부속서류` | 조회/삭제/부속서류가 동등 기능인지, 계층 구조인지 불명확 | 기능별 메뉴 분리: "신고서 조회", "제출한 신고서 삭제", "부속서류 제출" |
| `전자신고 접수증·납부서 출력` | 접수증과 납부서가 하나의 문서인지, 별도 출력인지 불명확 | "접수증 출력" + "납부서 출력" (분리) |
| `반복/복사 발급` | 비대칭 동사 구조 ("반복하다" ≠ "복사하다") | "반복 신고" + "신고서 복사" (기능 분리 또는 명확화) |

**이슈 6-C: `글로벌최저한세 신고` — 대상 독자 없음**

글로벌최저한세(Pillar Two, 15% 최저세율)는 다국적 대기업 CFO 전담 메뉴입니다.
현재 일반 납세자 메뉴 목록에 혼재되어 있으며 설명 링크도 없습니다.

**권고**: "세무대리·납세관리" 섹션으로 이동하거나, 법인세 하위 메뉴에 배치. 메뉴명 옆에 "(대기업 전용)" 또는 인라인 도움말 추가.

### 3.3 납부·고지·환급

**위치**: GNB > 납부·고지·환급 > [세부 메뉴]

#### 발견 이슈

**이슈 7: 표기 3종 불일치 (가운뎃점·슬래시·공백 혼용)**

같은 영역 내에서 구분자 표기가 일관되지 않습니다.

| 발견 표현 | 위치 |
|---------|------|
| 납부·고지·환급 (가운뎃점 ·) | GNB 대분류 |

**권고**: 가운뎃점(·)을 기준 구분자로 통일. 슬래시·쉼표·및 사용 금지.  
→ 상세 기준은 6장 "표기 통일 가이드" 참조.

**이슈 8: 카드 버튼 레이블 불일치**

납부 방법 선택 카드의 버튼 레이블이 맥락별로 다릅니다.

| 현재 | 사용 위치 | 문제 |
|------|---------|------|
| 납부하기 | 카드 A | — |
| 세금 납부 | 카드 B | 동사 아님, 명사형 |
| 납부 진행 | 카드 C | 진행 = 모호한 동작 |
| 바로납부 | 카드 D | 붙여쓰기 오류 |

**권고**: 모든 납부 시작 버튼 → "납부하기" 통일

**이슈 9: 미수령/미열람 행정어**

| 현재 | 문제 | 권고 |
|------|------|------|
| 미수령 고지서 | 수령(受領) 한자어 | 아직 받지 못한 고지서 |
| 미열람 통지서 | 열람(閱覽) 한자어 | 아직 확인하지 않은 통지서 |
| 미납부 | 납부 부정형 | 납부하지 않은 세금 |
| 체납액 조회 | 체납(滯納) 한자어 | 밀린 세금 조회 |

**접근 권고**: 뱃지/상태 태그 형태로 사용 시 짧게 유지가 불가피하다면
"미수령" 유지하되 인접 텍스트로 "아직 받지 못한 고지서"를 병기.

**이슈 9-B: 카드 UI 버튼 레이블 불일치 (CEO 실사 기준)**

| 카드 | 현행 버튼 | 문제 |
|------|---------|------|
| 납부할 세금 | `납부하기` | 동사형 ✅ |
| 미수령 환급금 | `환급금 수령 신청하기` | 길이 2.5배, 명사+동사 혼합 |
| 전자고지 미열람 | `열람하기` | 목적어 없음 — "무엇을" 열람? |
| 전자송달 | (버튼 없음) | 다른 카드와 구조 불일치 |

**권고**: 모든 카드 버튼 → `[납부하기]` `[환급금 신청하기]` `[고지서 확인하기]` `[신청하기]` (카드 타이틀이 목적어를 이미 제공)

**이슈 9-C: 이용시간 표기 복잡도**

| 현재 | 권고 |
|------|------|
| `이용안내 (365일) 06:00 ~ 24:00 단, 납부는 07:00 ~ 23:30` | `이용시간: 매일 06:00–24:00 (납부는 07:00–23:30)` |

변경 포인트: "(365일)" 제거 (매일과 동의어), 물결표(~) → 단대시(–), "단" 제거 후 괄호로 예외 처리

### 3.4 상담·불복·제보

**위치**: GNB > 상담·불복·제보 > [상담/불복청구/고충민원/심판청구]

#### 발견 이슈

**이슈 10: "불복청구" — 세법 전문 용어 노출**

불복청구(不服請求)는 행정심판법 용어입니다. 일반 납세자가 "불복청구를 하고 싶다"고 생각하지 않습니다.  
사용자의 실제 니즈: "세금 부과가 잘못됐다고 생각한다. 다시 따져보고 싶다."

| 현재 | 권고 |
|------|------|
| 불복청구 | 이의신청·심판청구 |
| 이의신청서 제출 | 이의신청서 제출 (유지 가능) |
| 심판청구서 작성 | 심판청구서 작성 (유지, 법령 용어) |
| 불복청구 진행상황조회 | 이의신청 진행 상황 조회 |

**이슈 11: "고충민원" 행정어**

| 현재 | 문제 | 권고 |
|------|------|------|
| 고충민원 | 고충(苦衷) — 한자어, 민원과 중복 의미 | 세금 관련 고충 신청 |
| 고충민원신청서 | 복합 명사 | 고충 신청서 |
| 민원 처리 결과 조회 | 유지 가능 | — |

**이슈 12: 30자 초과 메뉴명**

| 현재 메뉴명 | 글자 수 | 권고 |
|-----------|--------|------|
| 심판청구 및 소송관련 결정문 통지 등 | 19자 | 결정문·판결문 통지 |
| 세금 관련 범칙사건 처분결과 조회 | 18자 | 범칙처분 결과 조회 |
| 조세불복 처리 진행상황 안내 | 14자 | 조세불복 처리 현황 |

**이슈 12-B: 메뉴명 괄호 의미 불명확**

| 현재 | 문제 | 권고 |
|------|------|------|
| `탈세 및 각종 위반행위 제보(신고)` | 괄호 내 "(신고)"가 "제보"의 동의어인지, 별도 기능인지 불명확 | `탈세·법령 위반 제보` |
| `현금영수증·신용카드 미발급/발급거부` | 30자, 슬래시(/) 혼용 | `현금영수증 발급 거부 신고` |
| `세법해석 신청` | 대상 사용자 불명확 (전문가용) | `세법 의미 질의` + "(세무사·세무대리인 대상)" 부연 |

**이슈 13: 감정 플로우 안내 부재**

불복청구·고충민원 화면은 사용자가 **세금 부과 결정에 억울함·당혹감을 느끼는 상황**입니다.
현재 화면은 순수 서식 작성 안내만 있으며, 사용자의 감정 상태를 인정하는 문구가 전혀 없습니다.

**CEO 실사 확인**: 메뉴 클릭 → 로그인 유도 (아무 안내 없음). 기한 안내도 없음.

```
현재 플로우:
불복청구 클릭 → 로그인 화면 (이유, 기한, 절차 안내 전혀 없음)

권고 플로우:
불복청구 클릭 → "세금 부과에 이의가 있으신가요?
  불복청구는 고지일로부터 90일 이내에 신청해야 합니다.
  지금 어떤 상황인지 확인해 보세요." → [기간 확인하기] → 로그인
```

**권고**: 불복 절차 진입 화면 상단에 공감 문구 + 기한 안내 + 절차 안내 추가.
→ 상세 기준은 7장 "감정적 고부담 플로우 전용 어조" 참조.

### 3.5 장려금·연말정산·기부금

**위치**: GNB > 장려금·연말정산·기부금 > [세부 메뉴]

#### 발견 이슈

**이슈 14: UI 텍스트가 버튼 역할을 대신**

| 현재 | 문제 | 권고 |
|------|------|------|
| `근로장려금 신청기간: 5.1 ~ 6.30` (텍스트) | 클릭 불가 — 신청 버튼이 별도 | "근로장려금 신청하기" 버튼에 기간 병기 |
| `연말정산 간소화 자료 조회` (텍스트) | 텍스트가 링크인지 버튼인지 불분명 | `role="button"` 또는 명확한 버튼 컴포넌트로 교체 |

**이슈 15: 띄어쓰기 불일치**

| 현재 (불일치 사례) | 표준 |
|-----------------|------|
| 근로장려금 / 근로 장려금 혼용 | 근로장려금 (고유명사 형태로 통일) |
| 연말정산신고 / 연말정산 신고 혼용 | 연말정산 신고 (띄어쓰기 적용) |
| 기부금공제 / 기부금 공제 혼용 | 기부금 공제 |

**이슈 14-B: UI 가이드 텍스트가 콘텐츠 영역에 혼재**

```
실제 발견 문구:
"좌측의 근로·자녀장려금 메뉴버튼을 클릭하시면
 근로·자녀장려금 통합화면으로 이동 합니다."
```

문제:
- UI 조작 방법을 사용자에게 설명하는 텍스트가 콘텐츠 영역에 배치 — 버튼/링크가 있어야 할 자리
- "이동 합니다" — 띄어쓰기 오류 ("이동합니다"가 표준)
- "메뉴버튼" — 붙여쓰기 오류 ("메뉴 버튼")

**권고**: 해당 문구 삭제. 대신 명확한 버튼/카드 링크로 대체.

**이슈 15-B: 형용사 서비스명 (`편리한 연말정산`)**

| 현재 메뉴명 | 문제 | 권고 |
|-----------|------|------|
| `편리한 연말정산` | 형용사 서비스명 — 다른 메뉴(연말정산간소화, 근로소득 원천징수)와 형식 불일치 | `간편 연말정산` 또는 서비스 공식명칭 사용 |
| `전자기부금영수증` | 4자 복합 명사 | `전자 기부금 영수증` (띄어쓰기) |
| `연말정산간소화` | 공식 서비스명이나 띄어쓰기 필요 | `연말정산 간소화 서비스` |

**이슈 16: 장려금 신청 자격 안내의 정보 과부하**

장려금 신청 자격 조건이 4가지 요건(소득·재산·가구원·사업여부)을 단순 나열.  
사용자는 "내가 받을 수 있는지"가 핵심 니즈임.

**권고**: "3분 만에 확인하기" 형태의 자격 자가진단 위젯 텍스트 선행 제공.
안내문 예시:
```
✅ 개선:
근로장려금을 받을 수 있는지 먼저 확인해 보세요. (3분 소요)
[자격 확인하기]

자격이 확인되면 바로 신청할 수 있습니다.
```

### 3.6 구조적 이슈 (전역)

**전체 페이지에 걸쳐 발생하는 UX Writing 구조 문제**

**이슈 17: URL 리다이렉트 안내 없음**

`www.hometax.go.kr` 접속 시 `hometax.go.kr`로 리다이렉트되나 안내 없음.  
세무 대리인이 클라이언트에게 링크를 공유할 때 URL 혼란 발생.

**권고**: 404/리다이렉트 화면에 명확한 안내 추가.

**이슈 17-B: 로그인 필요 페이지 — 목표 URL 미보존**

현재 로그인 없이 특정 URL 직접 접근 시:
1. 메인 페이지로 리다이렉트
2. **리다이렉트 사유 표시 없음** — 사용자가 왜 메인으로 왔는지 알 수 없음
3. **목표 URL 보존 없음** — 로그인 성공 후 원래 페이지로 복귀 불가

```
권고:
"이 서비스는 로그인 후 이용할 수 있습니다.
로그인하면 원하시는 페이지로 바로 이동합니다."
[로그인하기]
```

기술 구현: `?redirect=/target-url` 파라미터 보존 + 로그인 성공 후 리다이렉트.

**이슈 18: 검색 placeholder 불일치**

| 발견 위치 | 현재 placeholder |
|---------|----------------|
| 홈 > 검색창 | "검색어를 입력하세요!" (느낌표 포함) |
| 공지사항 > 검색 | "검색어 입력" |
| 세금신고 > 내부 검색 | "찾고 싶은 내용을 입력하세요" |
| 도움말 검색 | "도움말 검색" |

**권고**: 전체 통일 → `"세금 종류·서비스명 검색"` (느낌표 제거, 구체적 힌트 제공)  
→ 상세 기준은 6장 "표기 통일 가이드" 참조.

## 4. 3대 원칙별 진단 요약

| 원칙 | 빈도 | 대표 위치 | 핵심 문제 |
|------|------|---------|---------|
| 무번역 위반 | ★★★ 매우 높음 | 세금 항목명, 불복 메뉴, 신고서 레이블 | 세법 전문 용어 그대로 UI 노출 |
| 정보핵심화 위반 | ★★★ 높음 | 14개 세금 비분류 나열, 장려금 자격 안내 | 사용자 맥락 없이 행정 정보 나열 |

| 표기 불일치 | ★★☆ 중간 | 구분자(·/,/및), 띄어쓰기, 버튼 레이블 | 동일 기능 다른 이름 |
| 접근성 | ★★☆ 중간 | 스크린리더 캡션, Hidden 칼럼 | SR 사용자 혼란 유발 |

## 5. Before / After 개선 예시

### E1 — 공지사항 상태 뱃지

```
🚫 현재:
COMING SOON  (영문)
NEW          (영문)
HOT          (영문)

✅ 개선:
준비 중
새 글
(HOT 뱃지 삭제 — 주관적 판단 금지)
```

> 적용 원칙: 무번역 (영문 → 한국어), 심리적안전망 (주관적 강조 표현 제거)

---

### E2 — 공지사항 테이블 스크린리더 칼럼명

```
🚫 현재 HTML:
<th>구분Hidden</th>

✅ 개선:
<th class="sr-only">유형</th>

<th aria-hidden="true"></th>  (시각적으로도 숨길 경우)
```

> 적용 원칙: 접근성 (WCAG 1.3.1 — 스크린리더 사용자에게 의미 있는 정보 제공)

### E3 — 세금신고 진입 화면 (사용자 유형 분기)

```
🚫 현재:
종합소득세신고·지방소득세신고
부가가치세신고
원천세신고
법인세신고
... (14개 나열)

✅ 개선:
어떤 상황에 해당하시나요?
[직장인·프리랜서]   [개인사업자]   [법인사업자]   [세무 대리인]

```

> 적용 원칙: 정보핵심화 (맥락 기반 필터링), 무번역 (역할어 사용)

### E4 — 신고 완료 화면

```

```

> 적용 원칙: 정보핵심화 (접수번호·납부기한 첫 줄), 무번역 (결정세액 → 납부할 세금)

### E5 — 납부 방법 선택 (카드 버튼 통일)


## 6. 표기 통일 가이드 (신설)

홈택스는 동일한 구분자를 페이지에 따라 다르게 사용합니다.  
이 섹션은 홈택스 내 표기 일관성 기준을 정의합니다.

### 6.1 구분자 기준

**홈택스 실사에서 발견된 3중 불일치** (같은 메뉴명, 다른 문자):

| 문자 | 유니코드 | 발견 위치 | 권고 |
|------|---------|---------|------|
| `·` | U+00B7 | GNB 탭 | ✅ 기준 문자 |
| `ㆍ` | U+318D | h1 제목, 페이지 타이틀 | ❌ 사용 금지 |
| ` · ` | U+0020+U+00B7+U+0020 | 브레드크럼 | ❌ 사용 금지 (공백 포함) |

**전체 구분자 기준표**:

| 구분자 | 용도 | 사용 여부 |
|--------|------|----------|
| · (가운뎃점, U+00B7, 공백 없음) | 메뉴명·레이블 내 항목 병렬 | ✅ 기본 구분자 |
| ㆍ (U+318D, 한국어 가운뎃점) | — | ❌ 사용 금지 |
| / (슬래시) | URL, 경로 표시 외 사용 금지 | ❌ 텍스트 구분자로 사용 금지 |
| , (쉼표) | 문장 내 나열 (최대 3개) | ⚠️ 문장 안에서만 허용 |
| 및 | 공식 문서·법령 인용 시 | ⚠️ UI 레이블에는 사용 금지 |

**적용 예시**:

```
✅ 올바른 구분:
납부·고지·환급
이의신청·심판청구

❌ 잘못된 구분:
납부/고지/환급   (슬래시)
납부 및 고지    (및 — 레이블에서)
납부, 고지, 환급 (쉼표 — 레이블에서)
```

### 6.2 띄어쓰기 기준

| 유형 | 기준 | 예시 |
|------|------|------|
| 고유명사 뒤 일반명사 | 띄어쓰기 적용 | 연말정산 **신고**, 근로장려금 **신청** |
| 고유명사만으로 구성 | 붙여쓰기 유지 | 근로장려금, 연말정산 |
| 동사형 버튼 레이블 | 붙여쓰기 허용 | 신청하기, 납부하기 |
| 복합 명사 (4자 이상) | 가독성 우선 띄어쓰기 | 원천징수 신고서 (← 원천징수신고서 ✗) |

### 6.3 숫자·금액·날짜 표기

| 항목 | 기준 | 예시 |
|------|------|------|
| 금액 | 원화 + 쉼표 3자리 | 1,230,000원 |
| 날짜 (기한) | M월 D일 형식 | 5월 31일 |
| 날짜 (범위) | ~ 사용 | 5월 1일~6월 30일 |
| 퍼센트 | % (한글 '퍼센트' 금지) | 10% |
| 전화번호 | 하이픈 없이 | 126, 1566-1700 |

### 6.4 영문·영어 혼용 금지

| 현재 사용 | 권고 |
|---------|------|
| COMING SOON | 준비 중 |
| NEW | 새 글 (뱃지) 또는 제거 |
| HOT | 제거 |
| FAQ | 자주 묻는 질문 |
| PDF | PDF (유지 — 고유명사) |
| URL | 주소 (일반 문장) / URL (기술 문서) |

## 7. 감정적 고부담 플로우 전용 어조

홈택스에는 사용자가 **심리적으로 힘든 상황**에서 접근하는 화면이 다수 존재합니다.  
이 화면들은 일반 안내 화면과 다른 **감정 배려형 어조**를 적용해야 합니다.

### 7.1 고부담 플로우 분류

| 화면 유형 | 감정 상태 | 적용 어조 |
|---------|---------|---------|
| 불복청구·이의신청 | 억울함, 혼란, 분노 | 공감 선행 + 절차 안내 |
| 체납·밀린 세금 조회 | 부끄러움, 걱정, 압박감 | 판단 배제 + 해결 경로 제시 |
| 사망자 관련 조회 (상속) | 슬픔, 피로, 혼란 | 간결함 + 단계별 안내 |
| 가산세 발생 | 당혹감, 긴장 | 사실 + 행동 + 문의 |
| 세무조사 관련 | 두려움, 긴장 | 권리 안내 선행 |

### 7.2 고부담 플로우 어조 규칙

**규칙 1: 감정을 인정하는 문장을 먼저**

```
❌ 바로 절차 시작:
"이의신청서를 작성하려면 아래 서식을 선택하세요."

✅ 감정 인정 후 절차:
"세금 부과 결정이 잘못됐다고 생각하신다면, 다시 검토받을 권리가 있습니다.
이의신청이나 심판청구 절차를 안내해 드리겠습니다."
```

**규칙 2: 판단하지 않는 중립 언어**

```
❌ 판단 포함:
"납부하지 않으신 세금이 있습니다. 즉시 납부하세요."

✅ 중립 언어:
"납부 기한이 지난 세금이 있습니다.
[지금 납부하기]  [분납 신청하기]  [상담 요청하기]"
```

**규칙 3: 권리와 선택지를 먼저**

```
✅ 권리 안내:
"세금 부과 결정에 이의가 있으시면:
• 90일 이내 이의신청
• 90일 이내 심판청구 (이의신청 결과 불복 시)
• 행정소송 (심판청구 결과 불복 시)

지금 어떤 단계에 계신가요?"
```

**규칙 4: 상담 창구를 항상 제공**

모든 고부담 플로우 화면 하단에 반드시 포함:
```
도움이 필요하시면 국세상담센터(126)로 전화해 주세요.
평일 09:00~18:00 운영 | 무료
```

### 7.3 사망자 관련 조회 (상속세·상속 신고) 어조

상속 관련 화면은 사용자가 **최근 가족을 잃은 상황**임을 고려합니다.

```
✅ 상속세 신고 안내:
가족을 잃은 슬픔 속에서도 신고 의무를 준비하셔야 해서 죄송합니다.
단계별로 차근차근 안내해 드리겠습니다.

[1단계: 신고 대상 자산 확인하기]
(약 10분 소요)
```

**금지 표현 (상속 관련)**:
- "즉시", "신속히", "빠르게" — 압박감 조성
- "귀하의 의무입니다" — 판단적
- "고인의" — 직접 지칭 회피 권고 (필요 시 "돌아가신 분의")

## 8. 기관 맞춤 톤·매너 권고

### 8.1 톤 기준

| 상황 | 권고 톤 | 샘플 |
|------|---------|------|
| 세금 항목 안내 | T2 길잡이 + 중립 | "납부할 세금: 320,000원. 5월 31일까지 납부해 주세요." |
| 신고 완료 | T4 마침표 (절제) | "종합소득세 신고가 완료됐습니다. 접수번호: {N}" |
| 오류·기한 초과 | T1 해결사 | "신고 기한이 지났습니다. 지금 신고하면 가산세가 줄어듭니다." |
| 처리 대기 | T3 중계자 | "담당자가 신고 내용을 확인 중입니다. (처리 기한: 6월 30일)" |
| 불복·체납 | T1 해결사 + 공감 | "억울한 세금 부과라고 느끼신다면, 이의신청할 수 있습니다." |

### 8.2 홈택스 전용 규칙

1. **세금 용어 일상어화 + 도움말 병기**: 결정세액(ⓘ 최종 납부 세금) 형식 권고
2. **금액은 항상 명시**: "세금이 부과됩니다" ❌ → "추가 세금 35,000원이 붙습니다" ✅
3. **기한은 구체적 날짜로**: "기한 내에" ❌ → "5월 31일까지" ✅
4. **가산세 안내는 공포 조성 금지**: 사실(가산세 발생) + 행동(지금 신고) 세트로 제공
5. **국세상담센터 126** 모든 에러·고부담 플로우 화면에 포함
6. **상속·사망 관련 화면**: 공감 문구 선행 필수

## 9. 자가진단 체크리스트 — 홈택스 맞춤

### 공통 항목 (Layer 2 상속)

```
□ 세법 전문 용어 첫 등장 시 괄호 또는 ⓘ 도움말로 일상어 설명이 있는가?
□ 에러 메시지에 사용자를 탓하는 표현이 없는가?
□ 에러 메시지에 해결 방법 또는 문의 연락처가 포함됐는가?
□ 결론이 첫 문장에 있는가?
□ 이중 부정이 없는가?
□ 막연한 약속("곧", "빠르게", "신속히")이 없는가?
□ 버튼 레이블이 동사형인가? ("제출" → "제출하기", "납부" → "납부하기")
```

### 홈택스 전용 항목

```
□ 납부·환급 화면에 금액과 날짜(기한·입금일)가 명시됐는가?
□ 신고 완료 화면에 접수번호가 포함됐는가?
□ 가산세 경고 화면에 "지금 신고하기" 행동 링크가 있는가?
□ 세션 만료 시 임시 저장 여부를 안내했는가?
□ 모든 에러 화면에 국세상담센터(126) 연락처가 있는가?
□ 세금 항목명 옆에 금액이 병기됐는가? (텍스트만으로 이해 가능한가?)
□ 구분자가 가운뎃점(·)으로 통일됐는가? (슬래시·쉼표·및 혼용 없는가?)
□ 영문 표현(COMING SOON, HOT, NEW)이 한국어로 교체됐는가?
□ 고부담 플로우(불복·체납·상속)에서 감정 공감 문구가 선행했는가?
□ 사망자 관련 화면에 "즉시", "신속히" 등 압박 표현이 없는가?
□ 상속·불복 화면에 국세상담센터 연락처 + 운영시간이 포함됐는가?
□ 검색 placeholder에 느낌표(!)가 없는가?
□ 14개 세금 신고 항목 진입 전에 사용자 유형 분기가 있는가?
```

### 10.3 배너 이미지 alt 텍스트 과부하 ★★

#### H20 — "바로가기>" 꺾쇠 스크린리더 오독

### §12 전체메뉴 모달 UX Writing 이슈 (Cycle 3)

#### H26 — 탭 레이블 가운뎃점 기호 혼용

전체메뉴 탭 레이블에서 두 가지 가운뎃점 기호가 혼용:
- 일반 탭: `·` (U+00B7, 반각 가운뎃점) — `"계산서·영수증·카드"`, `"납부·고지·환급"`
- 2번째 탭: `･` (U+FF65, 전각 가운뎃점) — `"증명･등록･신청･ 사업장현황"`

```
🚫 현재:
탭 1: "계산서·영수증·카드" (U+00B7)
탭 2: "증명･등록･신청･ 사업장현황" (U+FF65 + 줄바꿈)

✅ 개선:
탭 2: "증명·등록·신청·사업장현황" (U+00B7 통일, 공백 제거)
또는: "증명·등록·신청" (사업장현황 제거, 너무 긴 탭 분리)
```

> 원칙: **A** 기호 일관성 — 동일 기호 통일 사용

---

#### H27 — 탭 레이블 강제 줄바꿈

`"증명･등록･신청･ 사업장현황"` 탭 텍스트가 DOM에서 두 텍스트 노드로 분리 (`text: "증명･등록･신청･"` + `text: "사업장현황"`). 이는 탭 버튼 내부에서 강제 줄바꿈이 발생함을 의미.

```
🚫 현재:
증명･등록
·신청･
사업장현황 (줄바꿈)

✅ 개선:
탭명 단순화 또는 약어 사용:
"증명·등록·신청" (사업장현황 → 별도 탭 또는 하위 메뉴)
```

> 원칙: **B** 탭 레이블 간결성 — 4개 키워드를 한 탭에 압축하지 않음

---

#### H28 — 검색창 placeholder 어조 불일치

메인 페이지와 전체메뉴 내 검색창 placeholder가 다른 어조:

| 위치 | placeholder |
|------|------------|
| 메인 GNB 검색 | `"검색어를 입력하세요!"` (느낌표) |
| 전체메뉴 내 검색 | `"검색어를 입력하세요."` (마침표) |

```
✅ 개선 (통일):
"세금·서비스 이름을 검색하세요"
(느낌표 제거, 예시 힌트 추가, 두 곳 동일 적용)
```

> 원칙: **A** 동일 기능 동일 어조, **B** 예시 힌트로 검색 유도

---

#### H29 — "레이어 팝업" 무의미 모달 제목

전체메뉴 모달의 h2가 `"레이어 팝업"`. 스크린리더 사용자가 전체메뉴 모달에 진입하면 "레이어 팝업 대화상자"가 낭독되어 이 팝업의 목적을 알 수 없음. 실제 제목 `h2 "전체메뉴"`는 그 아래 별도로 있음.

```
🚫 현재:
<dialog aria-label="레이어 팝업">
  <h2>레이어 팝업</h2>  ← 무의미
  <h2>전체메뉴</h2>   ← 실제 제목

✅ 개선:
<dialog aria-labelledby="allMenuTitle">
  <h2 id="allMenuTitle">전체메뉴</h2>
  (불필요한 "레이어 팝업" 헤딩 제거)
```

> 원칙: **A** 모달 제목 명확성 — 접근성 4.1.2 (이름, 역할, 값)
